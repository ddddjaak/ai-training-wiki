## 这个 skill 做什么

shipping-and-launch 是一个**交付前检查流程**——在方案发给客户之前，系统化地检查所有交付物是否齐备、验收条件是否满足、已知风险是否可控、回滚方案是否就绪。在 AE 场景下，这个 skill 的核心价值是**防止"方案发出去了但漏了关键文件/测试/前提条件"的交付事故**。

> **类比**：你准备出差——护照、机票、酒店预订、充电器、名片、演示材料。shipping-and-launch 是那个"出门前最后检查一遍"的清单——不会到了机场才发现护照没带。

**和内部开发 shipping 的区别**：内部开发的 shipping 关注的是"代码能不能上线"。AE 的 shipping 关注的是**"客户能不能独立验收并开始使用方案"**——包含 demo 材料、集成说明、客户环境要求。

---

## 什么时候用 / 什么时候不该用

### ✅ 应该用（AE 场景）

| 场景 | 为什么需要 shipping |
|------|-------------------|
| 方案文档、代码、测试都完成了，准备发送给客户 | 交付前最后检查——防止漏掉关键交付物 |
| 方案涉及多个交付物（固件、文档、测试报告、demo 脚本） | 多文件交付容易漏——清单确保不遗漏 |
| 客户需要在特定环境下做验收（如高低温箱、特定设备） | 环境要求必须明确——客户环境不满足时方案无法验证 |
| 方案曾被回滚或大幅修改——需要确认最新版本完整 | 版本混乱时清单帮你还原"当前应该交付什么" |

### ❌ 不该用

| 场景 | 为什么不该用 | 替代方案 |
|------|------------|---------|
| 方案还在开发中 | shipping 是终点检查，不是开发中检查 | 用 [spec-driven](./3-3-spec-driven-development.md) 跟踪进度 |
| 交付物只有一个文件（如改一行的固件） | 不需要完整清单 | 手动确认即可 |
| 内部测试发布 | AE shipping 的"客户环境检查"不适用 | — |

    - [ ] Demo 演示步骤文档

    以上交付物的状态如何？哪些已完成、哪些待完成？

你: 核心交付物全部完成。波形对比图和功耗对比表还在整理，
    明天上午能好。Demo 演示步骤文档没写——需要写吗？

AI: Demo 演示步骤文档——如果客户需要自己重复验收测试，
    建议写一个简版（不是完整文档）：
    
    ```markdown
    # 验收测试 Demo 步骤
    
    ## 所需环境
    - 温度记录仪样机 × 1
    - 高低温箱（-20°C ~ +85°C）
    - USB-TTL 串口模块
    - PC 上位机（或串口终端）
    
    ## 测试步骤
    1. 烧录固件 spi_lowspeed_v1.0.bin（ST-LINK 或 DFU）
    2. 连接串口（115200 8N1），确认能看到启动日志
    3. 常温（25°C）下运行 10 分钟，串口无 SPI 错误输出
    4. 设置高低温箱 -20°C，运行 2 小时，串口无 SPI 错误输出
    5. 升温至 +85°C，运行 2 小时，串口无 SPI 错误输出
    6. 循环 3 次
    7. 测试结束后，检查 RCC_CSR 寄存器——确认无 IWDG 复位标志
    
    ## 通过标准
    - 全程无 SPI 通信错误
    - 全程无系统复位
    ```

你: 好，我写。其他检查项呢？

AI: 继续 shipping 检查清单。

    ## 验收条件对照（对照 spec v2.0）

    | AC# | 条件 | 是否通过 | 证据 |
    |-----|------|---------|------|
    | AC-1 | 三温度点各 10,000 次零错误 | ✅ 通过 | 测试报告 §3.1 |
    | AC-2 | 采样率 ≥ 100 sps | ✅ 通过（实测 203 sps） | 测试报告 §3.2 |
    | AC-3 | 功耗增量 ≤ 20µA | ✅ 通过（实测 +8µA） | 测试报告 §3.3 |
    | AC-4 | FreeRTOS 任务调度无冲突 | ✅ 通过 | 测试报告 §3.4 |

    | 条件 | 是否满足 | 行动 |

    4. 交付邮件草稿

## AE 交付清单模板

每次方案交付前，逐项检查：

```markdown
## AE 方案交付检查清单

### 1. 交付物完整性
- [ ] 方案文档（含所有修订历史）
- [ ] 方案规格书（含客户签字确认版）
- [ ] 固件（bin + hex，版本号一致）
- [ ] 测试报告（含所有 AC 的测试证据）
- [ ] 客户集成说明（烧录步骤、环境要求、注意事项）
- [ ] Demo 演示步骤（如客户需要自行验证）

### 2. 验收条件
- [ ] 对照规格书逐条确认——所有 AC 已满足
- [ ] 每条 AC 有可追溯的测试证据
- [ ] 边界条件已验证（不只是 happy path）

### 3. 客户环境
- [ ] 客户具备验收所需的环境和设备
- [ ] 客户技术人员可独立执行验收
- [ ] 已提供客户不会操作的项目（如烧录工具）的替代方式

### 4. 交付沟通
- [ ] 交付邮件草稿准备（包含：交付物列表 + 验收方法 + 反馈期限）
- [ ] 确认交付后的跟踪责任人
- [ ] 确认客户的内部审批流程（如有）

### 5. 回滚
- [ ] 回滚步骤明确（含操作时间估算）
- [ ] 回滚不影响客户的正常业务

### 6. 版本一致性
- [ ] 所有文件标注同一版本号
- [ ] 文档中没有 TODO / 占位符 / 内部草稿标记
- [ ] 文件名和内容中的版本号一致
```

## 做对了的标准是什么

### ✅ 成功输出

| 标准 | 说明 |
|------|------|
| **交付物清单完整** | 每个文件标注状态（完成/待完成）——没有"好像有但不太确定" |
| **验收条件对照表每条有证据** | AC → 通过/未通过 → 测试证据引用——不是"应该都通过了吧" |
| **客户环境确认完成** | 客户能不能独立验收——而不是等着客户说"我没有 XX 设备" |
| **回滚方案就绪** | 如果方案有问题，客户多久能恢复——不是"应该不会出问题" |

### ❌ 失败输出

| 失败表现 | 为什么不行 | 怎么办 |
|---------|-----------|--------|
| "都好了，发吧"——没有逐项检查 | 没有清单 = 依赖记忆——记忆会漏 | 对着清单逐项确认，而不是凭感觉 |
| 客户环境没确认——"我以为他们有 XX" | "我以为"是交付事故 #1 原因 | 交付前和客户确认环境 |
| 没有回滚方案 | 方案出问题时客户干等——AE 交付最基本的安全网 | 补回滚步骤——即使简单到"刷回原固件" |

## 常见误区

### 误区 1："方案很小，不需要 shipping 检查"

**现象**：方案只改了 1 行代码——你觉得不需要走 shipping 流程。结果发给客户后发现固件版本号错了（是上一个版本的），客户烧进去后行为不对。

**正确做法**：shipping 清单的复杂度可以调整——小方案用小清单（5 项），但**不能不检查**。即使只改 1 行，交付物完整性（固件版本号、测试报告、客户环境）仍然需要确认。

### 误区 2："验收条件 spec 里写过了，不用再对"

**现象**：spec 里写了 5 条 AC，你记得"都测过了"，没有再逐条对照。后来客户发现 AC-3（功耗）其实没测——你只测了 AC-1 和 AC-2。

**正确做法**：shipping 检查时必须**逐条对照**——不是"记得测了"，而是"测试报告 §X.Y 有通过证据"。

### 误区 3："回滚方案是给大方案准备的"

**现象**：你觉得回滚方案是给那种"改了几十个文件、跨几个团队"的大方案用的。小方案不需要。

**正确做法**：回滚方案的复杂度匹配方案的复杂度。小方案的回滚可以是"烧回上一个版本的固件"——但这也要写成文字。客户遇到问题时，你希望他有个明确的回滚步骤，而不是慌张打电话给你。

## 进阶

### 交付邮件模板

AI 可以生成标准的交付邮件：

```
帮我生成一封交付邮件给客户（陈工），内容：
- 交付物：降 SPI 速率方案
- 包含文件：方案文档、固件 bin、集成说明
- 客户需要做：按集成说明烧录固件 + 在高低温环境中验证
- 我们建议的完成时间：2 周内
- 如有问题请联系：我（手机/微信）
```

### 交付后的跟踪

建议在交付邮件中指定跟踪节奏：

```
交付后第 3 天：邮件 brief check（"烧录顺利吗？有问题吗？"）
交付后第 1 周：正式 tracking（"验收进度如何？需要什么支持？"）
交付后第 2 周：如无反馈，升级——电话确认
```
