## 这个 skill 做什么

spec-driven-development 是一个先写规格再实现的纪律——**在写代码之前，先用自然语言或结构化格式定义"这个功能应该做什么、不应该做什么、怎么验证它做对了"**。然后 AI 按照这份规格来实现，实现完成后对照规格逐条验收。

> **类比**：spec-driven 是"餐厅厨房的标准化菜谱"。没有菜谱——每个厨师凭感觉做，同一道菜今天咸明天淡。有菜谱——每个人都能做出一致的味道。spec 就是你的代码菜谱：输入（食材）、处理步骤（火候）、输出（成品标准）、验收（味道检验）。

---

## 什么时候用 / 什么时候不该用

### ✅ 应该用

| 场景 | 为什么 |
|------|--------|
| 实现一个新功能或模块 | 规格帮你理清"到底要实现什么"——减少实现阶段的反复 |
| 交给 AI 实现但你不会逐行审查 | 规格定义了验收标准——你只需要对照验收标准验证，不需要读懂每一行 |
| 多人协作或交接任务 | 规格是团队共识的基准——"做完了"的定义不会再因人而异 |
| 涉及精度、时序、功耗等可度量指标的功能 | 这些指标天然适合放在规格里作为验收条件 |

### ❌ 不该用

| 场景 | 为什么不该用 | 替代方案 |
|------|------------|---------|
| 修复一个已知的明确 bug | bug 的"规格"就是"修复后这个问题不再出现"——不需要额外写规格 | [debugging-and-error-recovery](./3-11-debugging-and-error-recovery.md) |
| 5 行以内的简单函数 | 写规格的时间超过写代码的时间 | 直接实现 + 测试 |
| 探索性原型——还不确定要什么 | 规格需要确定的输入/输出/约束，原型阶段还不清楚 | [incremental-implementation](./3-7-incremental-implementation.md) |

## 怎么触发

### 触发方式

```
用 spec-driven 实现一个 I2C 命令队列模块——先帮我写规格，确认后再实现。
```

或更简单：

```
先写规格，再实现：我需要一个环形缓冲区（Ring Buffer）模块，用来缓存串口接收数据。
```

### 规格模板

AI 会产出一份包含以下要素的规格：

```markdown
## 功能规格: I2C 命令队列模块

### 1. 概述
一个非阻塞的 I2C 命令队列——允许主循环提交 I2C 传输请求（读/写），
在后台（I2C 中断中）逐个执行，完成后回调通知。

### 2. 输入/输出

| 输入 | 类型 | 说明 |
|------|------|------|
| `i2c_cmd_t` | struct | 命令描述：\&#123;设备地址, 读写方向, 数据缓冲区, 数据长度, 回调函数} |
| `i2c_queue_push(cmd)` | 函数调用 | 提交一个命令到队列 |

| 输出 | 类型 | 说明 |
|------|------|------|
| `QUEUE_OK` | 状态码 | 命令已入队 |
| `QUEUE_FULL` | 状态码 | 队列已满，命令被拒绝 |
| `QUEUE_ERROR` | 状态码 | 参数错误 |
| `callback(status)` | 回调 | 命令执行完成后调用，传入执行结果 |

### 3. 行为规则

1. 命令按 FIFO 顺序执行
2. 队列深度：默认可配置（建议 8）
3. 一个命令完成后，中断自动启动队列中的下一个命令
4. 如果 I2C 总线错误（NACK），当前命令以错误状态回调，队列继续执行下一个
5. 访问队列的操作必须是中断安全的（原子操作或临界区保护）

### 4. 约束

- 不阻塞主循环（所有 I2C 传输在中断中完成）
- 不使用动态内存分配（队列缓冲区静态分配）
- 队列操作时间复杂度 O(1)
- I2C 中断优先级 ≤ 4（不能阻塞更高优先级的定时器中断）
- 回调函数在 I2C 中断上下文中执行 → 回调中不能调用 HAL_Delay 或其他阻塞 I2C 的函数

### 5. 验收条件

- [ ] AC1: 单个命令入队后能在 I2C 总线上观察到正确的波形（逻辑分析仪）
- [ ] AC2: 连续入队 8 个命令（队列满），第 9 个返回 QUEUE_FULL
- [ ] AC3: 队列中第 3 个命令故意让从机 NACK（写入无效地址）→ 
        该命令回调收到错误状态 → 第 4 个命令仍正常执行
- [ ] AC4: 10ms 内连续出入队 100 个命令——无丢命令、无死锁、无内存泄漏
- [ ] AC5: 空队列时不再触发任何 I2C 中断操作

### 6. Not Doing（本次不实现）

- 不支持命令优先级（纯 FIFO）
- 不支持命令取消/替换
- 不支持多 I2C 总线（只支持 I2C1）
- 不在回调中做超时处理（由提交方在回调中判断）
```

> 💡 **规格的核心要素**：概述（是什么）→ 输入/输出（接口）→ 行为规则（怎么做）→ 约束（限制条件）→ 验收条件（怎么知道做对了）→ Not Doing（做什么/不做什么的边界）。

## 规格→实现→验收

规格确认后，AI 进入实现阶段：

```
AI: 规格已确认。按以下顺序实现：
    1. 数据结构（i2c_cmd_t、队列结构体）
    2. 队列操作（push/pop/peek）
    3. I2C 中断处理（命令执行 + 回调）
    4. 公共 API（i2c_queue_init / i2c_queue_push）
    
    开始实现。完成后对照验收条件逐条验证。
```

实现完成后，AI 自动对照规格做验收：

```markdown
## 验收报告

- [x] AC1: 单命令波形——`HAL_I2C_Master_Transmit_IT` 调用后 
         `I2C1_EV_IRQHandler` 中执行完成，逻辑分析仪确认波形正确 ✓
- [x] AC2: 队列满检测——`QUEUE_DEPTH=8`，第 9 次 push 返回 `QUEUE_FULL` ✓
- [x] AC3: 错误恢复——NACK 后 `i2c_queue.c:124` 执行 `HAL_I2C_DeInit+Init` 复位，
         回调收到 `QUEUE_ERROR_NACK`，队列继续执行下一个 ✓
- [x] AC4: 压力测试——100 个命令在 847ms 内完成，无丢帧 ✓
- [x] AC5: 空队列——`i2c_queue_pop` 返回 NULL 后 `I2C_IT_disabled` ✓

全部 5 项验收条件通过。
```

## 做对了的标准是什么

### ✅ 成功输出

| 标准 | 说明 |
|------|------|
| **规格包含验收条件** | 不是"I2C 命令队列"就完了——有可度量的验收条件（AC1-AC5） |
| **验收条件是"可观测的"** | 不是"代码质量好"——而是"第 9 个入队返回 QUEUE_FULL"/"逻辑分析仪确认波形正确" |
| **有 Not Doing** | 明确了本次不做什么——防止范围蔓延 |
| **实现满足所有验收条件** | AI 实现完成后逐条对照验收条件验证——不是"我觉得差不多了" |
| **验收失败的处理** | 如果有验收条件没通过，AI 明确指出哪条没通过、为什么、需要怎么改 |

### ❌ 失败输出

| 失败表现 | 为什么不行 | 怎么办 |
|---------|-----------|--------|
| 规格只有功能描述，没有验收条件 | "I2C 命令队列"不是规格——是一句话概括。没有验收条件无法判断"做完了吗" | 补充验收条件——至少 3 条，覆盖正常路径 + 异常路径 + 边界条件 |
| 验收条件不可度量 | "可靠通信"不可度量——发生一次 NACK 算不算不可靠？ | 把验收条件改成可度量的具体标准 |
| 实现完了但没有验收报告 | 你无法确认实现满足了规格 | 要求 AI 逐条对照验收条件出验收报告 |

## 常见误区

### 误区 1："写规格浪费时间——我不如直接让 AI 实现，不对再改"

**现象**：你说"帮我实现一个 I2C 命令队列"。AI 实现了。你跑了一下发现"但你没考虑中断安全"，AI 修改。又跑了一下"但队列满了怎么办"，AI 再改。三轮下来花的时间超过了先写规格。

**正确做法**：规格是对齐期望的工具——你和 AI 就"做完是什么意思"达成一致。没有规格，每一次"不对再改"都是在迭代对齐期望——只是隐式的、低效的。

### 误区 2："规格要写得很细——把所有实现细节都写进去"

**现象**：规格里写了"用数组实现、用 `__disable_irq()` 做临界区保护、用 `memcpy` 复制数据"。

**正确做法**：规格定义**行为**（What），不是**实现**（How）。"队列操作必须是中断安全的"是规格；"用 `__disable_irq()`"是实现。把 How 留给 AI——如果 AI 选的实现不好，你针对实现做 review，而不是在规格阶段就限定。

**判断标准**：如果规格里的一条可以有两种以上的实现方式——它是规格。如果只有一种——它可能是实现细节。

### 误区 3："AI 实现的代码如果能编译通过+测试通过=满足了规格"

**现象**：AI 实现后编译通过、测试通过。你满意了。但后来发现"每次 I2C 传输完成后回调没有在主循环上下文中调用，导致回调中的 FreeRTOS API 调用触发 assert"。

**正确做法**：验收条件应该覆盖这种隐性约束。如果你知道"回调将在中断上下文执行"，这应该在规格的约束部分出现——对应的验收条件要验证这个约束是否被满足。

## 进阶

### 规格的迭代

规格不是一次写完的——AI 实现过程中可能发现规格的模糊之处。处理方式：

```
AI: 规格中说"队列满时返回 QUEUE_FULL"，但没有说调用方应该怎么处理。
    建议：调用方可以轮询等待，或者我们提供一个非阻塞的 try_push 函数。
    你倾向哪种？

你: 加一个非阻塞的 try_push——返回 QUEUE_FULL 时调用方自己决定是等待还是放弃。
AI: 好，已更新规格并实现。
```

### 与 interview-me 配合

如果需求本身还不够清晰，先用 [interview-me](./3-1-interview-me.md) 澄清需求，产出清晰的描述 → 再用 spec-driven 把描述变成可验收的规格 → 再实现。

### 与 TDD 配合

规格的验收条件（AC1-AC5）可以直接转换为 TDD 的测试用例。AC 描述的是"应该怎么做"，测试代码验证"真的这么做"。

## 客户对接场景：为客户写方案规格

> spec-driven 在两种场景下使用方式有所不同：**内部开发**（规格指导内部实现）和**客户对接**（规格作为 AE 和客户之间的验收契约）。

### 内部开发 vs 客户对接

| 维度 | 内部开发 | 客户对接 |
|------|---------|---------|
| 规格目的 | 指导内部开发实现 | 作为 AE 和客户之间的验收契约 |
| 验收方式 | 编译通过 + 测试通过 | 客户签字确认 + 验收条件满足 |
| 核心风险 | 实现偏离规格 | 客户说"我以为会包括 XXX" |

### 客户对接场景下什么时候用

| 场景 | 为什么需要 spec-driven |
|------|----------------------|
| 方案方向定了，需要转化成可验收的规格 | 客户需要知道"什么叫做好了"——spec 是双方的验收标准 |
| 客户可能在验收时说"我以为会包括 XXX" | spec 明确标注了范围边界和 Not Included |
| 方案涉及性能指标（功耗、速率、精度） | 指标必须量化——"更低功耗"不是 spec，"功耗 ≤ 80µA"才是 |

### 客户对接场景下的规格关键要素

1. **量化指标**：所有性能指标必须量化——"更低功耗"→"功耗从 120µA 降至 ≤ 80µA"
2. **范围边界**：明确标注 Not Included——防止客户"我以为"
3. **客户环境约束**：客户需要什么设备/环境来验收
4. **签字确认**：spec 必须经过客户确认——不是内部文档
