## 这个 skill 做什么

interview-me 是一个结构化需求澄清工具——它不直接给你答案，而是通过**一问一答**逐步把模糊的想法变成清晰、可执行的任务描述。每一轮 AI 会问你一个问题，你回答后 AI 会 restate（复述）你的意思，确认无误后再问下一个问题。最后输出一份你确认过的完整需求描述。

> **类比**：interview-me 是你的"需求翻译官"。你脑子里有一张模糊的图（"这个芯片功耗太高，得优化一下"），翻译官不直接猜测你想怎么优化——它通过结构化的提问帮你把模糊的图拆解成清晰的组件：功耗瓶颈在哪？目标降多少？允许牺牲什么（性能/精度/响应时间）？

---

## 什么时候用 / 什么时候不该用

### ✅ 应该用

| 场景 | 为什么需要 interview-me |
|------|--------------------------|
| 需求描述太模糊（"优化一下功耗"、"加个功能"） | 模糊的需求不能被直接执行——需要先澄清 |
| 你知道想要什么结果，但不确定怎么做 | interview-me 帮你从结果倒推路径 |
| 多个利益相关方给了不同方向的输入 | interview-me 帮你理清冲突和优先级 |
| 新接手一个任务，但需求文档写得很简略 | interview-me 帮你补全遗漏的细节 |

### ❌ 不该用

| 场景 | 为什么不该用 | 替代方案 |
|------|------------|---------|
| 需求已经很清楚、有明确的验收条件 | 不需要澄清，直接用 spec-driven | [spec-driven-development](./3-3-spec-driven-development.md) |
| "这个函数怎么写"（很具体的实现问题） | interview-me 是澄清需求，不是帮写代码 | 直接让 AI 帮你实现 |
| 紧急 bug fix（"I2C 挂了，快修"） | 不需要需求澄清，需要 debug | [debugging-and-error-recovery](./3-11-debugging-and-error-recovery.md) |

## 怎么触发

### 触发方式

```
我现在有一个模糊的需求——芯片跑 FreeRTOS，低功耗模式下功耗还是太高，帮我澄清一下需求。
```

或更简短：

```
帮我理清这个需求：老板说"把 I2C 通信的可靠性提升一下"
```

AI 收到后会进入 interview 模式——一轮一个关键问题，逐步澄清。

### 互动流程

interview-me 的典型流程如下：

```
Phase 1: 范围澄清（Scope）
  → "你说的功耗优化——是指哪个工作模式？Run/Sleep/Stop/Standby？
     不同模式的优化方向完全不同。"
  
  你回答后，AI restate:
  → "确认：你需要优化 Run 模式下的功耗，目标从 12mA 降到 8mA 以下，
     Sleep 模式暂时不动。对吗？"

Phase 2: 约束澄清（Constraints）
  → "Run 模式降功耗通常有几个方向：降主频、关不用的外设时钟、
     用 LPUART 替代 USART、减少唤醒频率。哪些是你可以接受的？"

  你回答后，AI restate:
  → "确认：可以降主频（80MHz→16MHz 在非通信时段），
     不能换外设（USART1 是硬件固定的），
     不能改变 RTOS tick 频率（维持 1kHz）。对吗？"

Phase 3: 成功标准澄清（Success Criteria）
  → "降到 8mA 以下算成功——但这个值是在什么条件下测的？
     所有外设全开？还是只开 USART1 + I2C1？（不同测试条件功耗差很多）"

  你回答后，AI restate:
  → "确认：测试条件 = USART1(115200) + I2C1(100kHz) + GPIO 4路输入，
     定时器全部关闭，ADC 不使能。8mA 是在这个条件下的上限。对吗？"

Phase 4: 约束整合（Integration）
  → "基于前面的讨论，总结一下任务：
     
     【目标】STM32L4 Run 模式功耗从 12mA 降到 ≤8mA
     【约束】
     - 可降主频（非通信时段 80→16MHz）
     - 不可换外设/USART 波特率
     - 保持 RTOS 1kHz tick
     - 不影响 I2C 通信（100kHz 不能降）
     - 不影响 GPIO 中断响应（≤100μs）
     【测试条件】USART1 + I2C1 + 4 GPIO 输入，定时器/ADC 关闭
     【Not Doing】Sleep/Stop 模式优化、更换 MCU、修改硬件
     
     这个理解完整吗？有没有需要补充的？"
```

### 关键行为

interview-me 的核心行为是**restate**（复述确认）。每轮问答后，AI 会用自己的话重复你的意思——这是一个"校验点"。如果 restate 不对，立即纠正；如果 restate 对了，继续下一个问题。

---

## 做对了的标准是什么

### ✅ 成功输出

| 标准 | 说明 |
|------|------|
| **你的意图被准确确认** | AI 的 restate 准确反映了你的原意——每次 restate 你都能说"对" |
| **模糊地带被消除** | 原来 "优化一下功耗" → 现在有具体的数值目标、约束条件、取舍范围 |
| **下一步行动明确** | 访谈结束后你知道"接下来该做什么"——是写 spec？拆任务？直接实现？ |
| **没遗漏关键约束** | 你会说"这个限制我没说但 AI 问到了"——说明 interview 覆盖了隐性约束 |
| **Not Doing 被明确** | 不仅确认了要做什么，也明确了不做什么——防止范围蔓延 |

### ❌ 失败输出

| 失败表现 | 为什么不行 | 怎么办 |
|---------|-----------|--------|
| AI 的 restate 歪曲了你的意思 | interview 的核心是准确理解——如果 restate 不准，整个访谈无效 | 立刻纠正："不对，我的意思是 XX，不是 YY" |
| AI 问的问题太泛（"能说一下你的具体需求吗"） | interview-me 应该问具体的、引导性的问题，不是开放式提问 | 追问："给我几个可能的方向让我选，不要开放式提问" |
| 所有问题都是技术实现细节，没有先澄清范围 | 顺序反了——先定范围（做什么/不做什么），再定约束，最后才是实现 | "先别问实现细节——先帮我确认目标范围和约束" |

## 常见误区

### 误区 1："AI 问的问题我都想到过"

**现象**：AI 问的问题（"你要降到多少 mA？"）你觉得很明显——你心里早有答案。于是觉得 interview-me 浪费时间。

**正确做法**：重点不是"问题本身"而是**restate**。"把功耗降到 8mA"和"把 Run 模式在 USART+I2C 工作条件下的功耗降到 8mA"是两种完全不同的需求。AI 的 restate 把你脑子里的隐性约束显性化——**你的团队看到这份 restate 应该和你有相同的理解**。如果 restate 不够精确，继续补充直到精确。

### 误区 2："AI 问完一轮就可以了"

**现象**：AI 问了 4-5 个问题，得出了总结。你觉得"可以了"，就直接进入实现。

**正确做法**：interview 的最终产出是一个**需求描述**——把它发给你的同事或上级："这是我对这个需求的理解，你看对吗？"如果同事看了之后有不同理解，说明需求还不够清晰，需要继续 interview。

### 误区 3："需求清楚了就直接让 AI 实现"

**现象**：interview 完成后，你直接说"好，帮我实现"。跳过了规划和验证设计。

**正确做法**：interview 是第一步（需求澄清）。下一步取决于任务复杂度：
- 简单任务（< 一个文件）：直接实现
- 中等任务（3-5 个文件）：先 [planning-and-task-breakdown](./3-4-planning-and-task-breakdown.md)
- 复杂任务（跨模块）：先 [spec-driven-development](./3-3-spec-driven-development.md)

## 进阶

### 复杂需求的多次 interview

当需求涉及多个子领域时（如"既要优化功耗又要加新功能还要改通信协议"），考虑分多轮 interview——每轮聚焦一个子领域：

```
第一轮：先 interview 功耗优化的范围和约束
第二轮：再 interview 新功能的需求
第三轮：再 interview 通信协议改动的兼容性约束
最后：整合三轮的输出，检查交叉约束
```

### 与 planning-and-task-breakdown 配合

interview 产出的清晰需求 → 直接喂给 [planning-and-task-breakdown](./3-4-planning-and-task-breakdown.md)：

```
根据以下需求描述，拆解为可执行的任务：
[粘贴 interview 的输出]
```

### 与 idea-refine 配合

如果你的需求还处于"我有个想法但不确定值不值得做"的阶段，先用 [idea-refine](./3-2-idea-refine.md) 发散探索。确定方向后再用 interview-me 收敛到具体需求。

## 客户对接场景：澄清客户的外部需求

> interview-me 在两种场景下使用方式有所不同：**内部开发**（需求来自 PM/技术负责人）和**客户对接**（需求来自外部客户邮件/微信/口头描述）。

### 内部开发 vs 客户对接

| 维度 | 内部开发 | 客户对接 |
|------|---------|---------|
| 需求来源 | 内部 PM / 技术负责人 | 外部客户（邮件、微信、口头） |
| 问题特点 | 可复现、有具体错误码 | 模糊、情绪化、可能不是技术问题 |
| 核心挑战 | 把模糊的技术需求具体化 | 把客户的非技术描述翻译成可验证的技术问题 |

### 客户对接场景触发示例

**场景**：客户邮件说"你们芯片在高低温箱里跑几天就挂了"。

"挂了"可能 = 复位、死机、通信中断、功耗异常。interview-me 帮你逐步剥离：

```
客户说"挂了" → 追问现象 → 是复位 / 死机 / 通信中断？
有串口日志吗？→ RCC_CSR 复位标志位是什么？
是独立问题还是多个问题混在一起？
```

### AE 专用技巧

**技巧 1：客户说的"挂了"翻译成技术可能性**

| 客户表述 | 可能的含义 | 追问方向 |
|---------|-----------|---------|
| "芯片跑挂了" | 复位 / 死机 / 通信中断 / 外设不响应 | "有没有串口日志？RCC_CSR 寄存器的复位标志位是什么？" |
| "通信不稳定" | 偶尔失败 / 特定条件下失败 / 速率相关 | "失败频率？有错误码吗？抓过波形吗？" |
| "功耗太高" | 运行功耗 / 睡眠功耗 / 某个外设功耗 | "测的哪个模式？和 datasheet 对比差多少？" |

**技巧 2：区分"客户想解决的问题"和"客户描述的现象"**

客户说"我想要加一个外部看门狗"——这不一定是真正的需求。真正的需求可能是"系统偶尔死机，需要自动恢复"。外部看门狗只是客户自己想的"解决方案"。interview-me 要挖出**底层需求**。

**技巧 3：AI 的 restatement 确认**

每次澄清后 AI 会重述它理解的问题。你必须检查 restatement 是否正确——关键结论必须回到客户那里确认，不能直接在 AI restatement 基础上做方案。

### 常见误区（AE 特有）

| 误区 | 正确做法 |
|------|---------|
| "客户说什么我就记下来，直接当需求做" | 客户给你的通常是他们自己想的解决方案，不一定是真正的需求。追问"为什么" |
| "AI 已经帮我澄清了，不需要再和客户确认" | AI 的 restatement 是内部理解——关键结论必须回到客户处确认 |
| "interview-me 只用在第一次接触客户问题时" | 客户中途改需求、实现中发现新约束时也应再次触发 |
