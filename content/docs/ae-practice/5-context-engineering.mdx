## 这个 skill 做什么

context-engineering 解决一个核心问题：**AI 对你的项目一无所知，需要你告诉它什么信息才能不瞎编**。它不是"给 AI 更多信息"，而是"给 AI 正确的信息"——项目架构、硬件约束、命名规范、已有代码结构。加载了正确上下文后，AI 不再发明不存在的函数、寄存器配置和文件路径。

> **类比**：context-engineering 是给 AI 派"项目 onboarding"。就像一个新人入职，你不能直接丢一句"你帮我改一下 I2C 驱动"——你需要先告诉 TA：项目用 STM32L4、I2C1 连 EEPROM、初始化在 `bsp/i2c.c`、CubeMX 生成的文件在 `Core/` 下不要手动改。不对 AI 做 onboarding，它就会猜——然后编造出不存在的 API 和路径。

---

## 什么时候用 / 什么时候不该用

### ✅ 应该用

| 场景 | 为什么需要 |
|------|-----------|
| 第一次在一个项目中用 AI（或在新项目中用） | AI 对这个项目零知识——必须先加载上下文 |
| AI 频繁编造不存在的函数/文件路径 | 上下文缺失——AI 在"蒙" |
| 更换了项目（从项目 A 切到项目 B） | 两个项目的硬件平台、目录结构、命名规范可能完全不同 |
| AI 给出的建议用的 API 和你用的库不一致 | AI 默认用了它的训练数据里的 API，而不是你项目实际用的 API |
| 项目结构复杂，有多个模块和层次 | 没有上下文，AI 不知道模块间的关系 |

### ❌ 不该用

| 场景 | 为什么不该用 | 替代方案 |
|------|------------|---------|
| 同一个项目内的连续对话 | 上下文已在会话中——不需要重复加载 | — |
| 简单的一次性问答（"这个 C 语法是什么意思"） | 不需要项目上下文 | 直接问 |
| 通用编程问题（"volatile 关键字的作用"） | 不依赖项目上下文 | 直接问 |

## 怎么触发

### 上下文的三层加载模型

```
第一层：CLAUDE.md（项目级——静态）
  → 项目用什么 MCU、什么 IDE、目录结构、命名规范、禁止做的事
  → 写一次，每次会话自动加载

第二层：codegraph（代码结构级——动态）
  → 函数调用关系、符号定义、模块依赖
  → AI 在需要时自动查询（不需要你手动触发）

第三层：会话中的文件引用（文件级——按需）
  → 具体的 .c/.h 文件内容、日志片段、编译输出
  → 你在对话中明确告诉 AI "读这个文件"
```

### 第一层：CLAUDE.md（最重要）

CLAUDE.md 是 context-engineering 的核心。参考 M2 [2-3-claude-md-guide](../M2-能力扩展/2-3-claude-md-guide.md) 了解完整写法。这里只强调关键原则：

**CLAUDE.md 必须包含的信息**（按重要性排序）：

| 信息类型 | 为什么需要 | 示例 |
|---------|-----------|------|
| MCU 型号 | AI 需要知道寄存器映射、外设行为、HAL 版本 | `MCU: STM32L431RCT6, Cortex-M4` |
| 项目目录结构 | AI 知道去哪里找文件、新文件应该放哪 | `bsp/` = 板级驱动, `app/` = 业务逻辑, `Core/` = CubeMX 生成 |
| 编码规范 | AI 生成的代码不会违反团队规范 | 中断处理禁止 HAL_Delay、寄存器操作用 LL 库 |
| 禁止事项 | 防止 AI 做有害操作 | 不修改 CubeMX 生成的文件、不使用动态内存分配 |
| 关键外设配置 | AI 知道当前的硬件约束 | `I2C1: PA9(SCL)/PA10(SDA), 100kHz, 上拉 4.7kΩ` |

### 第二层：codegraph（自动）

在项目的 `.claude/mcp.json` 中配置好 codegraph 后，AI 在需要时会自动查询：

```
# 你不需要手动触发 codegraph——AI 自动在以下时机使用：

1. 你提到了一个函数名，AI 想知道它在哪定义的
   → AI 自动调 codegraph_search

2. AI 想了解某个函数的调用上下文
   → AI 自动调 codegraph_context

3. AI 需要追踪从 A 函数到 B 函数的调用路径
   → AI 自动调 codegraph_trace
```

codegraph 的作用是**让 AI 能看到代码结构**，而不是凭训练数据的记忆猜测。

### 第三层：会话中明确引用

对于一些 AI 不会自动读取的信息，你在对话中明确要求：

```
在分析这个问题之前，先读一下：
- bsp/i2c_eeprom.c（EEPROM 驱动的完整实现）
- Core/Src/stm32l4xx_it.c（中断向量表——我需要确认 I2C1 的中断优先级）
- .claude/settings.json（项目的 AI 权限配置）
```

## Before/After 对比

### Before（没有上下文）

```
你: 帮我在 I2C 驱动里加个超时重试机制

AI: 好的。在 I2C_WriteByte() 函数中添加以下代码：
    if (I2C_GetFlagStatus(I2C1, I2C_FLAG_TIMEOUT)) \&#123;  ← 不存在的 API
        I2C_ClearFlag(I2C1, I2C_FLAG_TIMEOUT);
        I2C_ResetBus(I2C1);  ← 不存在的 API
        return I2C_TIMEOUT;
    }
```

问题：AI 用了标准外设库的 API 名称（你用的是 HAL）、发明了不存在的 `I2C_ResetBus()`、不知道你的项目里 I2C 驱动的文件名和函数命名风格。

### After（有 CLAUDE.md + codegraph）

```
CLAUDE.md:
- MCU: STM32L431RCT6, 使用 STM32 HAL 库（不是标准外设库）
- I2C 驱动: bsp/i2c_eeprom.c, 函数命名 EEPROM_Read/EEPROM_Write
- I2C 初始化: bsp/i2c.c 中的 MX_I2C1_Init()
- 错误处理: HAL_I2C_GetError(&hi2c1) 检查错误类型

你: 帮我在 I2C 驱动里加个超时重试机制

AI: [先通过 codegraph 查了 bsp/i2c_eeprom.c 的现有函数]
    你的 EEPROM_Read() 已经有重试逻辑——但 HAL_I2C_Mem_Read 的第 6 个参数
    timeout 目前是 100ms。对于 AT24C02 的写入周期（最大 5ms）来说足够，
    但如果在 I2C 总线上有其他耗时操作（如大容量 M24M02 的页写入可达 10ms），
    100ms 可能不够。

    建议：
    1. 将 timeout 提取为 #define EEPROM_READ_TIMEOUT_MS 200
    2. 在 HAL_TIMEOUT 返回后加一次 I2C 状态检查，确认总线是否仍然繁忙
    3. 如果 HAL_BUSY，执行 HAL_I2C_DeInit + Init 复位

    要我具体实现吗？
```

差异：
- API 名称正确（`HAL_I2C_Mem_Read` 而不是 `I2C_GetFlagStatus`）
- 引用了项目中实际存在的函数（`EEPROM_Read`）
- 理解了现有的重试逻辑（没有再发明一遍）
- 建议符合硬件实际（AT24C02 写入周期、不同型号的差异）

## 做对了的标准是什么

### ✅ 成功输出

| 标准 | 说明 |
|------|------|
| **API 名称全部正确** | AI 引用的函数、宏、类型全部是你项目实际使用的（HAL/LL/RTOS） |
| **文件路径全部正确** | AI 建议的文件位置、修改的文件路径全部存在 |
| **硬件参数准确** | AI 引用的时钟频率、引脚、外设编号和你的实际硬件一致 |
| **编码风格一致** | AI 生成的代码遵循你 CLAUDE.md 中定义的命名规范和代码风格 |
| **不发明限制** | AI 在你的约束内工作——如果 CLAUDE.md 说"不使用动态内存"，AI 不会建议 `malloc` |

### ❌ 失败输出

| 失败表现 | 为什么不行 | 怎么办 |
|---------|-----------|--------|
| AI 使用了你没在用库的 API | 上下文没加载或不够——AI 在用训练数据的默认值 | 检查 CLAUDE.md 是否写明了使用的库和版本 |
| AI 引用了不存在的文件 | 项目目录结构未告知 AI | 在 CLAUDE.md 中加 `## 项目结构` 段 |
| AI 给出的硬件配置和实际不一致 | 外设配置未告知 AI | 在 CLAUDE.md 中加 `## 关键外设` 段 |
| AI 建议修改 CubeMX 生成的文件 | 未在 CLAUDE.md 的禁止事项中声明 | 在 CLAUDE.md 中加"不修改 Core/ 下 CubeMX 生成的文件" |

## 常见误区

### 误区 1："项目目录结构 AI 自己能看出来"

**现象**：你觉得 AI "进到项目目录后自己会看文件结构"。但 AI 不会主动探索你的项目结构（除非你让它读目录）。没有结构信息时，AI 会把文件放到它猜测的位置——通常是错的。

**正确做法**：在 CLAUDE.md 中明确写目录结构——至少标注"哪些目录对应什么功能、哪些目录不能手动改"。

### 误区 2："CLAUDE.md 越详细越好"

**现象**：把 HAL 库的完整 API 参考、STM32 参考手册的时钟树都写进 CLAUDE.md。结果 CLAUDE.md 太长（> 200 行），AI 实际使用时反而找不到关键信息。

**正确做法**：CLAUDE.md 是"项目特有的约束"——MCU 型号、目录结构、编码规范、关键外设配置、禁止事项。通用信息（HAL API 参考、寄存器定义）不需要写——AI 在训练数据里有，遇到不确定时它自己会查 codegraph 或 context7。

**篇幅建议**：50-100 行。

### 误区 3："加载了 CLAUDE.md 就不会出错了"

**现象**：写好了 CLAUDE.md，但 AI 仍然偶尔用错 API。你觉得 CLAUDE.md 没用。

**正确做法**：CLAUDE.md 解决的是"项目特有信息缺失"问题（MCU 型号、目录结构、规范），不完全解决"特定 API 用法模糊"问题。API 层的问题需要三层配合：
1. CLAUDE.md 告诉 AI 用 HAL 库（不是标准外设库）
2. codegraph 告诉 AI 项目里实际调用了哪些 HAL 函数
3. context7（MCP）让 AI 能查最新 HAL 文档确认 API 签名

单靠 CLAUDE.md 不够，三层都加载才算完整的 context-engineering。

## 进阶

### CLAUDE.md 迭代方法

CLAUDE.md 不是一劳永逸的。每当 AI 犯了一个上下文相关错误时，问自己：

> "如果 CLAUDE.md 里有这条信息，AI 还会犯这个错吗？"

如果答案是"不会"——把这条信息加进 CLAUDE.md。这是最高效的 CLAUDE.md 维护策略——**driver 是 AI 的错误，不是你的猜测**。

### 多项目上下文切换

如果你的日常涉及多个项目（如项目 A 用 STM32L4，项目 B 用 NRF52），每个项目的根目录下放各自的 CLAUDE.md。在项目 A 目录中启动 Claude Code 会自动加载项目 A 的 CLAUDE.md。

### 与 source-driven-development 配合

对于第三方库的 API 验证（如确认 `nrf_drv_twi_tx` 的正确参数），context-engineering 解决"项目级上下文"，[source-driven-development](./3-8-source-driven-development.md) 解决"第三方库 API 级准确性"。两者配合使用。
