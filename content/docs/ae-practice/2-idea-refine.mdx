## 这个 skill 做什么

idea-refine 把你的模糊想法精炼成一个**结构化的 one-pager**——包含 HMW（How Might We）问题陈述、多个可行方向、每个方向的优劣、关键假设、以及明确的 Not Doing（不做什么）。它的目的不是给你"最佳方案"，而是帮你从"我有个想法"走到"我知道有哪些路可以走"。

> **类比**：idea-refine 是你的"导航路线规划"。你说"我想从北京到上海"——导航不会直接选一条路，而是给你 3 条：高速（最快但收费）、国道（免费但慢）、高铁+租车（组合方案），每条标注时间/成本/风险。你根据约束选择。

---

## 什么时候用 / 什么时候不该用

### ✅ 应该用

| 场景 | 示例 |
|------|------|
| 新功能方向不明确 | "我们想在传感器节点上加 OTA 升级——但怎么做？走 BLE 还是走 Sub-1G？" |
| 多个方案各有优劣，不知道怎么权衡 | "低功耗方案 A 用 STOP 模式 vs 方案 B 用 Sleep 模式 + 降频——哪个更适合？" |
| 方案评审会前需要准备材料 | "明天要汇报，需要一个 one-pager 把方向讲清楚" |
| 技术选型困难 | "RTOS 选 FreeRTOS 还是 Zephyr？要不要上 RTOS？" |

### ❌ 不该用

| 场景 | 为什么不该用 | 替代方案 |
|------|------------|---------|
| 需求已经很明确，只需要澄清细节 | 不需要发散——需要收敛 | [interview-me](./1-interview-me) |
| 方向已定，只需要拆解任务 | 不需要探索方向 | [planning-and-task-breakdown](./4-planning-and-task-breakdown) |

## 怎么触发

```
我有一个想法但不知道怎么选方向：我们的传感器节点需要 OTA 固件升级能力。
约束：电池供电（CR2032）、BLE 5.0 已使能、Flash 256KB 用了约 180KB。
帮我精炼一下方向。
```

### AI 产出结构

```markdown
## Idea One-Pager: 传感器节点 OTA 固件升级

### HMW（How Might We）
如何在电池供电的 BLE 传感器节点上实现可靠的 OTA 固件升级，
同时保证升级失败后设备可恢复、用户无需拆机？

### 方向 A: BLE OTA（利用已有 BLE 链路）
- 优势 ✅: 不增加硬件成本（BLE 已存在）、用户手机即可发起升级
- 劣势 ❌: BLE 传输速度慢（~10KB/s），256KB Flash 需要 ~25 秒传输
          + 写入时间（~15 秒）≈ 40 秒。CR2032 峰值电流可能不够。
- 关键假设: BLE 在传输 256KB 期间连接不会断开（实际测试需验证）
- 风险: 升级中途断电 → 设备变砖（需要 Bootloader 双区保护）

### 方向 B: 外部 Flash + 双区切换
- 优势 ✅: 安全（新固件先写外部 Flash → 校验 → 再切换）、恢复可靠
- 劣势 ❌: 增加硬件成本（SPI Flash ~¥3）、增加 PCB 面积
- 关键假设: SPI Flash 不被现有 SPI 设备冲突（SPI1 已被 LCD 占用）
- 风险: 可能需改用 SPI2，引脚冲突需重新分配

### 方向 C: 增量差分升级（Delta OTA）
- 优势 ✅: 只传输固件差异部分（通常 5-20KB），传输时间短
- 劣势 ❌: Delta 算法复杂度高、需要已知基线版本、恢复逻辑复杂
- 关键假设: 固件差异部分 ≤ 20KB（需分析历史版本变化量）
- 风险: Delta 算法引入的额外 Flash 开销、恢复方案的复杂性

### 推荐
方向 A（BLE OTA + Bootloader 双区保护）— 利用已有 BLE 链路，加最小的
安全保证（双区），在成本和可靠性之间取平衡。

### Not Doing
- 不支持蜂窝网络 OTA（NB-IoT 模块未选型）
- 不支持增量差分升级（v1 MVP 不做）
- 不支持远程回滚（升级失败后本地回滚到上一个版本）
```

## 做对了的标准是什么

### ✅ 成功输出

| 标准 | 说明 |
|------|------|
| **有 HMW 问题陈述** | 把"加 OTA"翻译成了带约束的问题陈述 |
| **有 2-4 个可行方向** | 不是只有一条路——给出了多个值得评估的方向 |
| **每个方向有优劣 + 关键假设** | 不是罗列方案——每个方案有可验证的假设 |
| **有 Not Doing** | 明确了本次不做的范围——防止选择困难 |

---

## 常见误区

### 误区："AI 选的方向就是对的"

idea-refine 的推荐方向是综合分析——但 AI 不知道你团队的具体约束（"老板不喜欢 BLE 方案"、"供应商 X 的 SPI Flash 已经在用"）。**推荐是你的决策参考，不是你的决策**。你应该把 one-pager 拿去和团队讨论。

## 客户对接场景：为客户生成可选方案方向

> idea-refine 在两种场景下使用方式有所不同：**内部开发**（方案面向内部技术评审）和**客户对接**（方案面向外部客户，需要对比和选择感）。

### 内部开发 vs 客户对接

| 维度 | 内部开发 | 客户对接 |
|------|---------|---------|
| 方案受众 | 内部团队（技术评审） | 外部客户（需要对比和选择感） |
| 方案要求 | 最优技术路径 | 多个可选方向 + trade-off 对比 + 推荐 |
| 核心风险 | 方案技术不可行 | 只给客户一个方案——客户说"没有其他选择吗？" |

### 客户对接场景下什么时候用

| 场景 | 为什么需要 idea-refine |
|------|----------------------|
| 客户问题已澄清，需要生成多个可选方案方向 | 不能只给客户一个方案——要有对比，让客户有选择感 |
| 客户需求存在多个路径，每条路径有不同 trade-off | 你需要让客户理解"为什么方案 A 比方案 B 贵但更可靠" |
| 客户自己提的方案可能不是最优解 | idea-refine 可能挖出客户没想到的更优方案 |

### AE 输出要点

1. **至少 3 个方向**：不能只给一个方案
2. **每个方向的 trade-off**：成本/时间/风险/可靠性对比表
3. **推荐 + 理由**：推荐哪个方向 + 为什么
4. **"建议不做的事"**：排除哪些方向 + 为什么排除
