## 这个 skill 做什么

design-review 是一个**4 视角对抗审查引擎**——它从正确性（Correctness）、完整性（Completeness）、一致性（Consistency）、可行性（Feasibility）四个维度对设计进行对抗审查，找出设计中的缺陷。在 SE 场景下，这个 skill 的核心价值是**在实现之前发现设计问题——此时修复成本是设计阶段的 1/10 ~ 1/100**。

> **类比**：你画好了装修平面图。design-review 是那个拿着图纸走一圈的建筑监理——"这个门开的方向和旁边的电灯开关冲突了"、"图纸上标了承重墙的厚度但没标材料——施工队不知道用什么"、"水管走这里会和暖通管道交叉"。这些问题在图纸上发现，改一笔就行。施工后再发现，拆墙重新来。

**相关 SE skill**：
- `design-review`：4 视角设计审查——针对架构设计的对抗审查
- `se-review`：SE 综合审查——覆盖需求/架构/设计/测试的全面审查

## 什么时候用 / 什么时候不该用

### ✅ 应该用（SE 场景）

| 场景 | 为什么需要 design-review |
|------|------------------------|
| 架构设计完成后、详细设计开始前 | 关键节点——架构中的缺陷会在详细设计中放大 10 倍 |
| 设计涉及多个子系统或学科（硬件+软件+机械） | 跨边界的设计错误最容易被遗漏——每个子系统的人只看自己那部分 |
| 设计涉及功能安全或合规要求 | 安全设计缺陷可能无法通过认证——修改成本极高 |
| 设计做了重大变更后 | 变更可能引入新的不一致——需要重新审查 |

### ❌ 不该用

| 场景 | 为什么不该用 | 替代方案 |
|------|------------|---------|
| 设计还在草稿阶段，频繁修改中 | 审查需要相当稳定的设计基线——频繁变更的审查是浪费 |
| 改动极小且局限在一个模块内 | 不需要 4 视角审查——自检或 peer review 足够 | code-review 或 peer review |
| 项目时间极度紧迫且无安全/合规风险 | 审查本身有时耗——但如果审查对象有安全影响，这个理由不成立 |

## 视角 1: 正确性（Correctness）
**审查问题：设计是否满足所有需求？**

### ✅ 已满足的需求

| 需求 ID | 需求 | 设计对应项 | 状态 |
|---------|------|-----------|------|
| FR-1.1 | 3 路温度采集 | ADC1 scan 模式（IN1/IN2/IN16）+ DMA | ✅ 满足 |
| FR-2.1 | 预警级响应 | otpEngineTask 状态机 WARNING 状态 | ✅ 满足 |
| FR-2.2 | 降功率级响应 | otpEngineTask → TIM1 CCR 写入 | ✅ 满足 |
| FR-2.3 | 紧急关断级响应 | SW 路径（软件触发 BRK） + HW 路径（COMP1 → BRK） | ✅ 满足（双路径）|
| FR-3.1 | 硬件冗余检测路径 | COMP1 + DAC1 → TIM1 BRK | ✅ 满足 |
| FR-3.2 | 硬件路径独立于软件 | COMP1 输出内部直连 TIM1 BRK——不经过 GPIO、不依赖任何 ISR | ✅ 满足 |
| FR-4.1 | 故障记录到 EEPROM | otpEventTask → I2C1 → AT24C02 | ✅ 满足 |
| NFR-1 | 时序要求 | 各级延迟目标已在设计中定义 | ✅ 满足 |
| IR-1~5 | 接口需求 | 引脚分配表 + 外设选择已覆盖 | ✅ 满足 |

### ⚠️ 需求覆盖缺口

**COR-1: NFR-3（EMC 要求）的设计响应不够具体**

问题：架构设计说"NTC 信号应进行硬件 RC 低通滤波（截止频率 ~10 Hz）"。但：
- 实际设计的 LPF 截止频率是 15.9 kHz（R=100Ω, C=100nF）——和文档描述的 ~10 Hz 差了约 3 个数量级
- 10 Hz 截止频率需要 R=10kΩ, C=1.6µF（或 R=160kΩ, C=100nF）——这在输入端引入的阻抗问题需要评估

**严重度**：中 — ADC 输入阻抗 + 外部 R 的分压效应可能影响 NTC 测量精度

**建议**：明确实际使用的 LPF 参数并重新计算。如果使用 10kΩ 串联电阻 + 100nF 电容，截止频率 ~160 Hz，且需要确认 ADC 输入阻抗约束——根据 DS12288 Table 67，12-bit/2.5 ADC clock 下最大允许源阻抗 R_AIN_max = 100Ω（fast channel）或不可用（slow channel）。若使用 10kΩ 串联电阻，需大幅增加采样时间（如 640.5 ADC clock，此时 R_AIN_max ≈ 39-50kΩ）。

---

**COR-2: NFR-2（IEC 60730 Class B 自检）的运行时 PC 自检未设计触发机制**

问题：架构设计列出了 Class B 自检项（CPU 寄存器/PC/栈/ADC 参考电压），但没有说明：
- 运行时 PC 自检的触发频率（IEC 60730 要求至少每个主循环一次，或通过独立时基触发）
- 栈溢出检测的实现方式（硬件 MPU？软件 canary？FreeRTOS 自带栈检测？）
- ADC 参考电压监控的具体方式（与什么比较？偏差阈值？）

**严重度**：高 — 自检触发机制不明确 → 认证审计时可能被判定为"措施不完整"

**建议**：补充 Class B 自检的设计细节——至少标注每个检查的触发频率、检测方法、失败响应。

## 视角 2: 完整性（Completeness）
**审查问题：设计是否覆盖了所有场景和边界条件？**

### ⚠️ 遗漏的场景

**COM-1: 上电初始化的瞬态处理未设计**

问题：系统上电时，NTC 从室温开始升温。在 ADC 初始化完成但尚未获得第一个有效采样时——温度环形缓冲区是空的（或初始化为 0）——otpEngineTask 会读到什么？如果初始化为 0（即 0°C），所有阈值判断正常。但如果初始化为 0xFFFF（即 6553.5°C），会立即触发紧急关断。

**严重度**：中 — 可能导致产品首次上电误触发关断

**建议**：
1. 温度环形缓冲区初始化为 -273°C（或标记为 INVALID）
2. otpEngineTask 在检测到 INVALID 数据时跳过判断——直到至少 1 个有效采样
3. 增加初始化完成标志位 `otp_ready = false`，前 3 个采样周期（300ms）不触发保护

---

**COM-2: NTC 开路/短路故障的检测和处理未设计**

问题：NTC 是外部连接器元件——最易发生开路（连接器松脱）或短路（焊点桥接）故障。当 NTC 开路时，ADC 读到的是上拉电压（≈ VREF+ ≈ 3.3V），经 NTC 转换后相当于极低温度。**这意味着 NTC 开路会被系统误判为"温度正常"**——三级保护全部失效。

**严重度**：高 — 单点故障可能导致所有温度保护失效

**建议**：
1. 增加 NTC 合理性检查：ADC 读数应在 NTC 正常工作范围内（对应 -40°C ~ +150°C），超出范围视为传感器故障
2. NTC 开路检测（ADC 读数 > VREF+ - 50mV）和短路检测（ADC 读数 < 50mV）
3. 传感器故障时：LED 以特定模式闪烁（区别于三级保护）+ 通过 UART 上报 + **直接进入降功率模式**（安全侧处理——不确定温度时假设最坏情况）
4. 更新需求文档——补充传感器故障的需求项

---

**COM-3: EEPROM 写入期间掉电的异常处理未设计**

问题：紧急关断后 500ms 内写入 EEPROM（FR-4.1）。如果在这 500ms 窗口内系统电源跌落（电机紧急关断时反电动势可能回灌——电源环境恶劣），EEPROM 写入可能失败或写入一半——下次上电读到损坏的故障记录。

**严重度**：中 — 故障记录是事后分析的关键数据；损坏的记录失去价值

**建议**：
1. EEPROM 写入使用校验和（每个故障记录尾部附加 CRC-8）
2. 上电时检查故障记录 CRC——损坏的记录标记为 INVALID
3. 在电源上保持一个大电容（如 100µF 电解）确保关断后有足够能量完成写入（至少 1ms × I2C 写入时间）

## 视角 3: 一致性（Consistency）
**审查问题：设计内部各模块之间是否有冲突？**

### ⚠️ 不一致项

**CONS-1: 任务优先级分配与响应延迟要求存在冲突风险**

问题：
- NFR-1 要求紧急关断 SW 路径延迟 ≤ 50ms
- otpEngineTask 优先级 = 4（与 PWM 控制任务同级）
- 架构文档说"PWM 控制任务死循环时 HW 路径兜底"

但如果 PWM 控制任务在一个长临界区（关中断）中运行 60ms——otpEngineTask 同为 pri 4 无法抢占，SW 关断路径的 50ms 延迟目标将无法满足。

实际评估：
- 如果 PWM 控制任务的最大连续执行时间 ≤ 5ms，则 50ms 目标可达
- 如果 PWM 控制任务包含一个阻塞操作（如等待 SPI 传输 32 bytes @ 2MHz = 128µs）——不影响
- 如果 PWM 控制任务在关中断状态运行较长时间（如 Flash 页擦除——禁用中断 ~20ms）——50ms 窗口可能被突破

**严重度**：中 — 取决于 PWM 控制任务的实际最长阻塞时间

**建议**：
1. 测量 PWM 控制任务的最长连续执行时间（包括临界区）
2. 如果 > 10ms，建议 otpEngineTask 提升至 pri 5（高于 PWM 控制）——或拆分 PWM 控制任务的长临界区
3. 在设计中明确标注：SW 关断路径的 50ms 目标依赖于 PWM 控制任务的阻塞行为——需要实际测量确认

---

**CONS-2: ADC 采样周期与 FreeRTOS 时基精度不一致**

问题：
- FR-1.2 要求温度采集周期 ≤ 100ms
- 架构设计使用 FreeRTOS 软件定时器（100ms 周期）
- 但 FreeRTOS 软件定时器的精度受系统 tick 频率限制
  - 如果 configTICK_RATE_HZ = 1000（1ms tick），100ms 定时器的实际周期在 99-101ms 之间——可接受
  - 如果 configTICK_RATE_HZ = 100（10ms tick），定时器精度 ±10ms——边缘可接受
  - 如果 FreeRTOS 使用无 tick 模式（tickless idle），定时器精度取决于低功耗唤醒时钟

**严重度**：低 — 在典型配置下（1000Hz tick），波动在 1% 以内——不影响系统功能

**建议**：在架构文档的"假设"部分注明 configTICK_RATE_HZ 需 ≥ 1000。如系统使用 tickless 模式，温度采集应使用硬件定时器（TIM6/TIM7）而非 FreeRTOS 软件定时器。

## 视角 4: 可行性（Feasibility）
**审查问题：设计在给定 MCU 和硬件约束下是否可实现？**

### ⚠️ 可行性风险

**FEA-1: COMP1 非反相输入（PA1）与 ADC1_IN2（PA1）的引脚复用存在约束**

问题：架构设计将 PA1 同时用于 COMP1_IN+ 和 ADC1_IN2。这在 STM32G4 的硬件上是可能的——但是：
- COMP1 的输入可以使用 PA1，ADC1_IN2 也使用 PA1——这是允许的复用
- 但 COMP1_IN+ 需要一个内部连接到 COMP 的专用开关（COMP_C1CSR 寄存器中的 INPSEL 位（注意：SYSCFG_CFGR1 + COMP1_INP_SEL 是 STM32F3 架构，STM32G4 使用独立的 COMP_CxCSR 寄存器））
- 需要确认：COMP1_INP_SEL 设置为 PA1 时，不影响 ADC 对该引脚的采样

**实际验证**（查阅 STM32G4 参考手册 RM0440 第 24 章 Comparator 部分——注意 §21 为 ADC 章节）：
- COMP1 输入选择通过 COMP1_CSR 寄存器的 INPSEL 位配置
- 设置为 PA1（INPSEL=0）时，COMP1 的正相输入来自 PA1 的模拟开关（INPSEL 是 COMP_C1CSR 中的 1-bit 字段）
- ADC 采样 PA1 通过独立的 ADC 模拟开关——两者互不影响 ✅

**严重度**：已验证通过——但文档中应显式标注引脚复用兼容性，防止实现时误配置。另外注意：电机控制场景下，COMP1 可配置 BLANKING 字段（COMP_C1CSR 中）在 PWM 开关噪声期间抑制比较器输出——设计审查应确认是否需要使能 BLANKING 以降低误触发风险

---

**FEA-2: 故障记录缓冲区（30 个采样点 × 3 通道）的存储需求可能超出 AT24C02 容量**

问题：FR-4.1 要求记录关断前 1 秒内温度变化（每 100ms 一个点 × 3 通道 = 30 个数据点）。每个数据点：
- 通道 ×1（1B）+ 温度 ×1（2B, int16_t × 0.1°C）+ 时间戳偏差 ×1（1B, 0-9 表示 100ms 偏移）= 4B/点
- 30 数据点 = 120B

一次关断事件的完整记录：
- 事件头（编号 2B + 时间戳 4B + 触发通道 1B + 触发温度 2B + RCC_CSR 1B）= 10B
- 30 个采样点 = 120B
- CRC-8 = 1B
- 单次事件总计 = 131B

AT24C02 容量 = 256B。**只能存储 1 次完整关断事件**。

**严重度**：高 — FR-4.1 要求记录"关断事件"（隐含多次），但 EEPROM 空间只能存 1 次

**建议**：
1. 方案 A（推荐）：改用 AT24C08（1KB）或 AT24C16（2KB）——可存储 7-15 次关断事件，引脚兼容无需改 PCB
2. 方案 B（不改硬件）：只记录最近 1 次关断的完整数据 + 前 N-1 次的摘要（事件编号 + 时间戳 + 触发温度，每条约 8B）
3. 方案 C：压缩历史数据——不存每 100ms 的完整曲线，只存每 500ms 的采样点

---

**FEA-3: otpEventTask 在紧急关断后需要执行 131B 的 I2C 写入，但系统可能在 500ms 窗口内掉电**

问题：架构设计要求紧急关断后 500ms 内完成 EEPROM 写入。但：
- I2C1 @ 400kHz（Fast Mode）写入一个故障记录：需 17 次页写入（AT24C02 页大小 8B），含各次 Start/Addr/ACK/Stop 开销后总线时间约 3.95ms，加上 17 × 5ms 页写入周期 = 总计约 89ms
- AT24C02 每页 8B，页写入周期 5ms/页：131B = 17 页 × 5ms = 85ms
- 总计约 88ms 写入时间——在 500ms 窗口内绰绰有余 ✅

但是——紧急关断意味着 MOSFET 桥关闭 → 电机惯性旋转 → 反电动势回灌到电源 → VDD 可能在 50-200ms 内跌破 MCU 工作电压（取决于反电动势保护电路和电容储能）

**严重度**：高 — 如果 VDD 在写入完成前下跌，EEPROM 记录可能不完整或损坏

**建议**：
1. 硬件：评估 VDD 保持电容需求——若 MCU 写入期间功耗 25mA，VDD 从 3.3V 跌至 2.7V，100µF 电容仅能维持约 2.4ms。维持 200ms 需约 8300µF（不实际）。实际方案：减少写入数据量（先写 8B 摘要 + CRC），或使用超级电容 / 备用电源
2. 软件：紧急关断事件先写入 1 条简短的"摘要记录"（事件编号 + 时间戳 + 触发温度 = 8B，约 6ms 完成），剩余的详细曲线数据延后写入
3. 增加"写入完成标志"——上电时检查上次写入是否干净完成

## 4 视角审查详解

### 视角 1: 正确性（Correctness）

**核心问题**：设计是否满足所有需求？

逐条需求对照设计——检查是否有需求在设计中被遗漏、是否有设计元素没有对应任何需求（过度设计）。

### 视角 2: 完整性（Completeness）

**核心问题**：设计是否覆盖了所有场景和边界条件？

特别关注：
- 上电/断电瞬态
- 传感器故障模式（开路/短路/漂移）
- 资源耗尽的退化行为（队列满/EEPROM 满/栈溢出）
- 并发场景（多任务同时触发保护）

### 视角 3: 一致性（Consistency）

**核心问题**：设计内部各模块之间是否有冲突？

特别关注：
- 接口定义中的命名/类型/单位不一致（模块 A 输出 °C，模块 B 期待 ADC 原始值）
- 时序约束冲突（模块 A 要求 50ms 响应延迟，但模块 A 依赖的模块 B 执行时间 > 50ms）
- 资源共享冲突（两个模块同时需要 DMA1_Channel1）

### 视角 4: 可行性（Feasibility）

**核心问题**：设计在给定硬件和约束下是否可实现？

特别关注：
- 引脚复用冲突
- 存储/内存容量
- 实时性约束在给定 MCU 频率下是否可达
- 外设功能是否支持设计的用法（如 COMP 的内部连接）

## 做对了的标准是什么

### ✅ 成功输出

| 标准 | 说明 |
|------|------|
| **4 个视角都产出了审查意见** | 不是只审查了"看起来对不对"——完整性、一致性、可行性各有发现 |
| **每个发现标注了严重度** | HIGH/MEDIUM/LOW——方便排修复优先级 |
| **每个 HIGH 以上发现给出了修复建议** | 不只是"这里有问题"——附带"可以这样修" |
| **正确性视角逐条追溯了需求→设计** | 不是凭感觉说"设计看起来正确"——逐条需求核对设计中有没有对应项 |

### ❌ 失败输出

| 失败表现 | 为什么不行 | 怎么办 |
|---------|-----------|--------|
| 审查报告全是"设计良好，无问题" | 任何系统设计都有问题——"无问题"说明审查不够深入 | 换一个视角重新审——或者换一个审查者 |
| 发现的问题多是格式/命名问题 | 设计审查应该找设计缺陷——不是拼写检查 | 加具体约束："只关注影响系统行为的问题" |
| 审查发现了 30 个问题但没分类/没优先级 | 30 个未分类问题 = 开发团队无视这份报告 | 标注严重度 + 必须修复 vs 建议修复 |

## 常见误区

### 误区 1："设计审查就是找 bug——我自己设计不会有 bug"

**现象**：你是这个架构的设计者，你也来做审查。你审查自己的设计——自然觉得"都对"。

**正确做法**：设计审查必须由**非设计者**执行——或者至少 AI 用对抗式 prompt 执行。人的大脑会自然为自己的设计辩护——AI 的对抗审查可以穿破这个偏见。

### 误区 2："HIGH 发现太多了，设计要全部重做"

**现象**：审查报告出了 4 个 HIGH——你感觉设计被"否掉了"。其实每个设计审查都会找出一批问题——正常的架构设计在实现之前有 5-15 个发现（含 HIGH 和 MEDIUM）是健康状态。

**正确做法**：HIGH 项需要修复——但不意味着设计失败。修复 HIGH 项后重新审查——第二轮应该大幅减少发现数。

### 误区 3："可行性视角不重要——反正硬件已经定了"

**现象**：你觉得硬件（MCU、引脚、存储芯片）是现成的——可行性不是问题。但审查发现了 AT24C02 容量不够存完整故障记录——这是个真实的硬件约束。

**正确做法**：可行性视角的价值恰恰在于**在硬件已定的约束下验证设计**——不是在理想硬件上的理想设计。

## 进阶

### 与 se-review 配合

design-review 聚焦于架构设计。如果需要更全面的审查（需求 + 设计 + 测试 + 追溯），用 se-review。

### 与 traceability-matrix 配合

审查中发现的需求遗漏 → 反哺需求文档 → 更新 [traceability-matrix](./4-7-traceability-matrix.md)。

### 审查后的行动

审查报告 → 修复 HIGH/MEDIUM 项 → 重新审查（至少验证 HIGH 是否已修复）→ 进入详细设计阶段。修复过程本身也可以交给 AI——给 AI 审查报告中的某个发现，让它生成修复方案。
