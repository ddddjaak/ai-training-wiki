## 这个 skill 做什么

code-static-review 是一个**编码规范合规审查引擎**——它对照指定的编码标准（MISRA-C、CERT C、公司内部规范）检查代码，找出违反规则的代码行。在 SE 场景下，这个 skill 的核心价值是**确保护代码在"写对"之前先"写规范"**——编码规范违规可能不会导致今天的 bug，但会在明天的维护中导致 bug。

> **类比**：功能测试检查"这辆车能开吗"——刹车、油门、转向正常。code-static-review 检查"这辆车的零件符合规格吗"——每个螺丝的扭矩是否正确、焊点是否达标、线束是否绑扎规范。螺丝扭矩不达标今天可能不会出问题——但在颠簸路上行驶 10000 公里后就会松脱。

**和 M3 code-review 的区别**：
- M3 的 [code-review-and-quality](../ae-practice/9-code-review-and-quality)：5 轴审查（正确性+可读性+架构+安全+性能）——关注代码的功能和质量
- M4 的 code-static-review：**仅关注编码规范的合规性**——不关注功能是否正确。这是 SE 视角：代码是否遵守了组织定义的编码标准

**相关 SE skill**：
- `code-static-review`：编码规范审查——对照 MISRA/CERT C/自定义规范检查违规
- `se-review`：SE 综合审查——可以包含编码规范审查作为其中一个环节

## 什么时候用 / 什么时候不该用

### ✅ 应该用（SE 场景）

| 场景 | 为什么需要 code-static-review |
|------|------------------------------|
| 代码提交后、功能测试前 | 编码规范问题应在最早阶段拦截——不要让不规范代码进入集成测试 |
| 项目需要满足功能安全标准（IEC 60730, ISO 26262, IEC 61508） | 这些标准强制要求编码规范合规——是认证审计的一部分 |
| 新成员加入团队——需要确认代码符合团队规范 | 新成员可能不了解团队的特定编码约定 |
| 代码重构后 | 重构可能引入新的规范违规（如变量命名、代码结构） |

### ❌ 不该用

| 场景 | 为什么不该用 | 替代方案 |
|------|------------|---------|
| 代码还在频繁修改中（草稿阶段） | 每改一次就审一次规范——浪费。等内容基本稳定后再审 | 完成功能逻辑后再触发审查 |
| 代码是自动生成的（CubeMX 生成代码） | 自动生成代码不需要审查编码规范（生成器本身应保证合规） | 只审查手写代码部分 |
| 需要在审查中同时评估代码逻辑正确性 | code-static-review 只看规范——不看逻辑是否正确 | 用 M3 [code-review](../ae-practice/9-code-review-and-quality) |

## 违规详情（共 10 条）

### STATIC-1 [MISRA-C:2012 Rule 10.4 — Required]
**位置**: otp_engine.c:82
**违规**: 算术运算操作数的实质类型不匹配

```c
// 当前代码：
int16_t temp_raw = adc_read_channel(0);
uint16_t offset = 100;
int16_t temp_corrected = temp_raw + offset;  // int16_t + uint16_t
```

**问题**: `temp_raw` 的实质类型是 signed，`offset` 的实质类型是 unsigned。MISRA 10.4 要求算术运算的两个操作数具有相同的实质类型——signed 和 unsigned 在 C 的 usual arithmetic conversions 下会产生非预期的隐式转换结果。

**修复**:
```c
int16_t temp_corrected = temp_raw + (int16_t)offset;
```

---

### STATIC-2 [MISRA-C:2012 Rule 14.4 — Required]
**位置**: otp_engine.c:156
**违规**: 控制表达式应使用实质布尔类型（Rule 14.4）

```c
// 当前代码：
int16_t temp_adc_raw = adc_buffer[0];
if (temp_adc_raw) \&#123;  // temp_adc_raw 是 int16_t，不是布尔
    ...
}
```

**问题**: MISRA 14.4 要求控制表达式（if/while/for 的条件）必须是"实质布尔类型"。`temp_adc_raw` 是 `int16_t`——不是布尔类型。

**修复**:
```c
if (temp_adc_raw != 0) \&#123;
```

---

### STATIC-3 [MISRA-C:2012 Rule 17.7 — Required]
**位置**: otp_engine.c:198
**违规**: 函数 `otp_check_threshold` 的返回值未被使用

```c
// 当前代码：
otp_check_threshold(channel, THRESHOLD_WARNING);  // 返回值被丢弃
```

**问题**: `otp_check_threshold` 声明为 `otp_status_t` 返回类型，但在此调用点返回值被忽略。MISRA 17.7 要求函数的返回值必须被使用（除非显式转换为 void）。

**修复**:
```c
(void)otp_check_threshold(channel, THRESHOLD_WARNING);
// 或:
otp_status_t status = otp_check_threshold(channel, THRESHOLD_WARNING);
if (status != OTP_OK) \&#123; /* 错误处理 */ }
```

---

### STATIC-4 [MISRA-C:2012 Rule 15.5 — Advisory]
**位置**: otp_engine.c:220-296
**违规**: 函数 `otp_state_machine` 包含多个返回点（76 行内 5 个 return）

```c
// 当前代码（简化）：
void otp_state_machine(void) \&#123;
    if (!otp_ready) return;             // return 1
    if (check_shutdown()) return;       // return 2
    if (...)
        return;                          // return 3
    ...
    if (...) \&#123;
        ...
        if (...)
            return;                      // return 4
    }
    return;                              // return 5
}
```

**问题**: MISRA 15.5 要求函数有且仅有一个返回点（Advisory 级别，但在功能安全相关代码中通常升级为 Required）。

**修复**: 使用 goto 集中返回（在 MISRA 允许 goto 向前跳转到函数末尾的模式）——或重构条件逻辑让函数只有一个出口。

**注意**：此规则对代码结构影响较大——如果团队认为 5 个 return 更可读，可以申请豁免（记录在豁免清单中）。

---

### STATIC-5 [MISRA-C:2012 Rule 8.7 — Advisory]
**位置**: otp_engine.c:50
**违规**: 外部链接的全局变量应在仅一个翻译单元引用时限制为文件作用域 `temp_ringbuf` 应使用指针操作而非直接访问

```c
// 当前代码：
temp_sample_t temp_ringbuf[TEMP_RINGBUF_SIZE];  // 全局变量
```

**问题**: MISRA 8.7 建议将仅在一个翻译单元引用的函数和对象定义为内部链接（static）。`temp_ringbuf` 是全局变量——虽然在嵌入式系统中很常见，但应通过访问函数封装。

**修复建议**（Advisory——可根据团队实践决定）:
```c
static temp_sample_t temp_ringbuf[TEMP_RINGBUF_SIZE];  // 限制作用域为本文件
// 提供访问函数：
const temp_sample_t* otp_get_sample(uint8_t idx);
```

### STATIC-6 [MISRA-C:2012 Rule 12.1 — Advisory]
**位置**: otp_engine.c:67
**违规**: 表达式中的运算符优先级依赖

```c
// 当前代码：
if (temp & 0xFF == THRESHOLD_MASK) \&#123; ... }
```

**问题**: `&` 的优先级低于 `==`。当前表达式实际解析为 `temp & (0xFF == THRESHOLD_MASK)`——而不是预期的 `(temp & 0xFF) == THRESHOLD_MASK`。这是 MISRA 12.1 明确禁止的（依赖运算符优先级）。

**修复**:
```c
if ((temp & 0xFF) == THRESHOLD_MASK) \&#123; ... }
```

**⚠️ 注意**：这不只是一个"规范问题"——当前代码的行为可能与预期不符。如果 `THRESHOLD_MASK` ≠ `0xFF`，`(0xFF == THRESHOLD_MASK)` 为 0（false），则 `temp & 0` 恒为 0——条件永远不会为真。这是一个**实际的功能 bug**，在编码规范审查中暴露。

### STATIC-7 [CERT C INT30-C]
**位置**: otp_engine.c:134
**违规**: 无符号整数回绕

```c
// 当前代码：
otp_event_counter++;  // otp_event_counter 是 uint16_t
```

**问题**: 如果 `otp_event_counter` 到达 65535 后继续递增，将发生无符号回绕（wrap to 0）。CERT C INT30-C 要求检测或防止无符号整数回绕。

**修复**:
```c
if (otp_event_counter < UINT16_MAX) \&#123;
    otp_event_counter++;
} else \&#123;
    // 达到最大计数——不再递增（事件编号饱和）
    // 或者重置为 0 并记录"计数器回绕"事件
}
```

### STATIC-8 [公司规范: 函数体 ≤ 100 行]
**位置**: otp_engine.c:88-198
**违规**: 函数 `otp_evaluate_thresholds` 共 110 行——超过 100 行限制

**建议**: 将函数拆分为：
- `otp_evaluate_single_channel()` — 单通道阈值判断 (~40 行)
- `otp_evaluate_all_channels()` — 遍历所有通道 + 整合结果 (~40 行)
- `otp_determine_action()` — 根据结果决定响应动作 (~30 行)

### STATIC-9 [公司规范: 禁止动态内存分配]
**位置**: otp_event.c:45
**违规**: 使用了动态内存分配

```c
// 当前代码：
otp_event_t* event = pvPortMalloc(sizeof(otp_event_t));
```

**问题**: 公司嵌入式 C 规范明确禁止动态内存分配——即使 `pvPortMalloc` 是 FreeRTOS 提供的，仍视为动态分配。原因是：嵌入式系统需要可预测的内存行为——动态分配可能导致碎片化或分配失败。

**修复**: 使用静态预分配：
```c
static otp_event_t event_pool[MAX_PENDING_EVENTS];  // 静态事件池
// 使用简单的"使用/空闲"标记管理事件池
```

## 编码规范审查的核心价值

### 不只是"代码格式审查"

初学者容易把 code-static-review 等同于"检查缩进和命名"。实际上——编码规范审查经常发现真实 bug：

| 规范类型 | 审查时检查什么 | 常见发现 |
|---------|---------------|---------|
| 类型安全 (MISRA 10.x) | 类型转换、位运算操作数 | 隐式截断、符号扩展错误 |
| 控制流 (MISRA 14.x/15.x) | if/while 条件、返回点 | 优先级错误、逻辑漏洞 |
| 资源管理 (CERT C MEMxx) | 动态分配、栈使用 | 内存泄漏、栈溢出风险 |
| 整数安全 (CERT C INTxx) | 溢出/回绕/截断 | 计数器回绕导致逻辑错误 |

### SE 审查 vs 开发人员自审

开发人员自审代码时关注"逻辑对不对"——很少逐条对照 MISRA。SE 的 code-static-review 填补这个缺口——SE 关注代码是否满足**组织级别的质量标准**。

## 做对了的标准是什么

### ✅ 成功输出

| 标准 | 说明 |
|------|------|
| **每条违规标注了规则编号和位置** | 开发者能直接跳到对应行——不需要"在某处有个问题"的模糊描述 |
| **每条违规给出了修复建议代码** | 不只是"这里违规了"——附带"改成这样就合规了" |
| **违规按严重度分类** | 功能影响 > 安全影响 > 可读性影响 |
| **豁免建议附带理由和风险评估** | 有些规则在某些场景下不适用——SE 需要评估并记录豁免 |
| **审查覆盖了所有声明的标准** | 声明审查 MISRA + CERT C + 公司规范——三条都要有发现或通过 |

### ❌ 失败输出

| 失败表现 | 为什么不行 | 怎么办 |
|---------|-----------|--------|
| 只列出了违规行号但没有上下文 | 开发者要逐行跳到代码文件看——浪费时间 | 每行违规附带代码片段 |
| 把 MISRA Advisory 当成 Mandatory 要求修复 | 过度审查——团队花时间修复不必须修的东西 | 区分规则级别——Advisory 可接受豁免 |
| 审查 300 行代码只找到 1 个违规 | 除非代码是 MISRA 专家写的——否则 300 行业余代码通常有 8-20 条违规 | 确认审查标准是否配置正确——或换更全面的规则集 |

## 常见误区

### 误区 1："编码规范审查是 CI 的事——不需要 AI"

**现象**：你用 PC-Lint 或 Coverity 做静态分析——觉得 AI 的 code-static-review 没有价值。

**正确做法**：AI 和传统静态分析工具是互补的——不是替代关系。传统工具擅长**语法级别的规则检查**（如"MISRA 10.1: 操作数类型不一致"）。AI 擅长**语义级别的模式识别**——比如发现"这个 return 值的忽略可能在 3 层调用栈后导致未初始化变量被使用"——工具做不到这种上下文推理。

### 误区 2："豁免就是绕过规则——尽量不免"

**现象**：你觉得 MISRA 的规则应该全部遵守——申请豁免等于"接受不完美"。

**正确做法**：编码规范是**通用规则**——在某些嵌入式场景下确实会产生不必要的约束。MISRA 本身设计了"Deviation"（偏离）机制——豁免不是绕过规则，而是**在理解规则意图的前提下做出有记录的有意偏离**。关键是记录理由和风险评估。

### 误区 3："先写功能再管规范——规范是最后一步"

**现象**：你先把功能代码写完了，所有测试通过了，再跑 code-static-review。发现有 15 条违规——其中 3 条需要重构函数结构——改动会影响已通过的测试。

**正确做法**：编码规范审查应在功能测试之前进行。更理想的做法：在 CLAUDE.md 或项目配置中预设编码规范——让 AI 在生成代码时就遵守规范，而不是生成后再审查。

## 进阶

### 与 CLAUDE.md 配合

在项目的 CLAUDE.md 中预设编码规范——让 AI 在生成代码时就遵守：

```markdown
## Coding Standards
- MISRA-C:2012 compliance required (Mandatory + Required rules)
- No dynamic memory allocation (static pools only)
- Function body ≤ 100 lines
- Variable naming: snake_case
- All function return values must be checked
- Use explicit type casts for all implicit conversions
```

### 与 CI/CD 配合

将 code-static-review 集成到 CI 流程——每次代码提交流水线自动触发审查。AI 的审查报告可以作为传统 SAST 工具的补充。

### 审查豁免的管理

建立项目的豁免清单文件（deviations.md），记录所有活跃的规则豁免：

```markdown
| ID | 规则 | 文件/函数 | 豁免理由 | 风险评估 | 审批日期 |
|----|------|----------|---------|---------|---------|
| DEV-001 | MISRA 15.5 | otp_state_machine() | 5 个 early return 更可读 | 低 | 2026-06-17 |
| DEV-002 | MISRA 8.7 | temp_ringbuf | 全局环形缓冲区——嵌入式系统合理使用，限制为 static 文件作用域 | 低 | 2026-06-17 |
```
