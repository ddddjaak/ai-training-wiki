## 核心价值

| 没有权限管理意识 | 有权限管理意识 |
|---------------|-------------|
| AI 每次写文件都弹确认框 → 你机械点 "Yes" → 失去了对危险操作的敏感度 | 提前配置好：读什么都可以，写只限 `src/`，删文件必须确认 |
| 团队新人 clone 项目 → AI 可以 `sudo` → 安全隐患 | 项目级 `.claude/settings.json` 锁定危险命令 → 新人开箱即用且安全 |
| AI 要访问外网查 datasheet → 每次都要点确认 → 打断心流 | 提前 allow 外网查询 → AI 自由搜索，但写入仍需你确认 |

> **一句话**：权限管理不是"限制 AI"——是让你把确认精力花在真正危险的操作用，而不是每次都机械点 Yes。

---

## 全景位置

```
settings.json（规则配置） → Permissions（执行拦截） → 你的确认（最后防线）
       ↓                           ↓
  定义什么可以/不可以          每次工具调用前检查规则
```

Claude Code 的权限模型是 **默认只读，写入需批准**。你可以通过配置把信任的操作设为自动允许，把危险操作设为坚决阻止。

---

## 三层权限

| 级别 | 含义 | 使用场景 |
|------|------|---------|
| **allow** | 自动允许，不弹确认框 | `Bash(npm run *)` — 运行脚本是安全的 |
| **deny** | 坚决阻止，AI 无法执行 | `Bash(sudo *)` — 永远不允许 sudo |
| **ask**（默认） | 每次弹出确认框 | `WebFetch` — 拉外网内容让你看一眼 |

---

## 配置在哪里

三个 Settings 文件，作用范围从大到小：

| 文件 | 位置 | 作用范围 | 典型用途 |
|------|------|---------|---------|
| `settings.json`（用户级） | `~/.claude/settings.json` | 你的所有项目 | 个人偏好：允许 `npm run`、`git log` |
| `settings.json`（项目级） | 项目 `.claude/settings.json` | 团队共享（提交到 git） | 团队规范：禁止 `sudo`、允许 `src/**` |
| `settings.local.json` | 项目 `.claude/settings.local.json` | 仅你自己 | 个人工具路径、本地 MCP 配置 |

> **优先级**：local > 项目级 > 用户级。local 文件不提交 git（在 `.gitignore` 中）。

## 怎么配置

### 方式一：用 `/permissions` 交互式配置

```
/permissions
```

进入权限管理界面后，你可以：
- 查看当前所有 allow/deny 规则
- 添加新规则
- 将某个规则从项目级移到用户级
- 导入/导出规则

### 方式二：直接编辑 settings.json

```json
\&#123;
  "permissions": \&#123;
    "allow": [
      "Bash(npm run *)",
      "Bash(git:*)",
      "Bash(cmake:*)",
      "WebSearch",
      "WebFetch",
      "Read(**/*.md)",
      "Edit(src/**)"
    ],
    "deny": [
      "Bash(sudo *)",
      "Bash(rm -rf *)",
      "Bash(curl * | sh)",
      "Read(.env*)",
      "Read(**/secrets/**)"
    ]
  }
}
```

> ⚠️ **注意**：`Read(.env*)` deny 比 `Read(**/*.md)` allow 优先级更高——deny 永远胜出。

### 方式三：CLI 启动参数

```bash
# 临时信任模式（仅当前会话）
claude --dangerously-skip-permissions

# 仅允许特定目录
claude --add-dir /path/to/project
```

## AE 工程师推荐配置

```json
\&#123;
  "permissions": \&#123;
    "allow": [
      "Bash(git:*)",
      "Bash(npm:*)",
      "Bash(make:*)",
      "Bash(cmake:*)",
      "Bash(arm-none-eabi-*)",
      "Bash(openocd:*)",
      "Bash(jlink:*)",
      "WebSearch",
      "WebFetch(domain:st.com)",
      "WebFetch(domain:github.com)",
      "Read(**/*.c)",
      "Read(**/*.h)",
      "Read(**/*.md)",
      "Read(**/*.s)",
      "Read(**/*.ld)",
      "Edit(src/**)",
      "Edit(include/**)"
    ],
    "deny": [
      "Bash(sudo *)",
      "Bash(rm -rf *)",
      "Bash(curl * | sh)",
      "Bash(> /dev/*)",
      "Read(.env*)",
      "Read(**/secrets/**)",
      "Edit(**/*.bin)",
      "Edit(**/*.hex)"
    ]
  }
}
```

> **关键点**：
> - `allow` 了交叉编译工具链（`arm-none-eabi-*`）和调试器（`openocd`/`jlink`）——这些是 AE 工程师的高频命令，每次都确认太烦
> - `deny` 了固件二进制文件（`.bin`/`.hex`）的编辑——这些只能由编译生成，不能手动改
> - `WebFetch` 限制了域名——只允许 st.com 和 github.com，防止 AI 访问不可信的站点

## 对 AE 工程师最重要的权限策略

| 策略 | 为什么 | 怎么配置 |
|------|--------|---------|
| **外设寄存器操作需确认** | 写寄存器可能改变硬件状态（电压、时钟）——不可逆 | `Bash(*write*register*)` → ask |
| **允许编译、禁止烧录** | 编译是安全的，烧录到硬件需人工确认（可能覆盖产线固件） | `Bash(openocd *flash*)` → ask |
| **允许读 datasheet、禁止随意下载** | 查资料安全，但 AI 下载的二进制工具链可能有风险 | `WebFetch` → allow，但 `Bash(wget *)` → ask |
| **禁止修改链接脚本和启动文件** | `.ld` 和 startup 文件错了会导致整个固件无法启动 | `Edit(**/*.ld)` → ask 或 deny |

## 常见误区

| 误区 | 为什么错 | 正确做法 |
|------|---------|---------|
| "全 allow 了省事" | 你失去了对危险操作的最后一道防线——而且会养成不看确认框直接点 Yes 的习惯 | 只 allow 你理解且信任的操作。如果不确定，保持 ask |
| "deny 了 AI 就不会尝试" | deny 之后 AI 仍可能尝试——它只是执行时被拦截。浪费 token 和时间 | 在 CLAUDE.md 中同时声明"不要使用 XX"——双重保障 |
| "项目级 settings.json 就够了" | 项目级会提交 git——个人工具路径（如 `C:\Users\yufan\tools\`）不该出现在团队配置中 | 个人路径放 `.claude/settings.local.json` |
| "权限配置一次就完了" | 新工具、新 MCP、新 skill 都会引入新的权限需求 | 定期检查：每季度 review 一次 allow/deny 规则 |

## 常见卡点

| 现象 | 自救方案 |
|------|---------|
| 某个命令每次都弹确认框 | `/permissions` → 找到对应规则 → 设为 allow。或者直接说 "allow make commands" |
| 配置了 allow 但不生效 | 检查是否有 deny 规则覆盖了（deny 优先级最高）→ 检查文件名匹配（glob pattern 可能不对）→ 重启 Claude Code |
| 团队成员的权限配置被我的覆盖了 | 确认 `.claude/settings.local.json` 在 `.gitignore` 中——否则 local 设置会污染团队 |
| 不确定某个操作该 allow 还是 deny | 原则：可逆的、不涉及外部系统的 → allow。不可逆的、影响硬件的、涉及网络的 → ask/deny |
| CC Switch 换了 API 供应商后权限失效 | 不同供应商的 API 可能有不同的工具可用范围——检查 `/status` 确认当前模型和可用工具 |
