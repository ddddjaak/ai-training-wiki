## 核心价值

**Before**：你每次改完代码后手动跑 `make lint`、手动检查是否改了不该改的文件、手动确认 commit message 格式。这些事不难，但你偶尔会忘——比如提交完才发现没跑静态分析，或者改了生成文件（`main.c` 是 CubeMX 生成的）不该手动改。

**After**：Hooks 让 AI 在特定时机自动执行检查——构建后自动跑 lint、修改 CubeMX 生成文件时自动警告、提交后自动检查 commit message 格式。你不再需要"记得"做这些事——hooks 是你的自动化安全带。

> **类比**：Hooks 是 IDE 的"文件保存时自动格式化"功能。你不需要每次手动按 Ctrl+Shift+F——编辑器在保存时自动帮你做。Hooks 一样——你不需要每次手动做重复性的检查，AI 在正确的时机自动帮你做。

## 怎么触发

Hooks 通过 `.claude/settings.json` 配置。你不需要手动触发——AI 在满足条件时自动执行。

### Hooks 基本概念

| 概念 | 说明 | 类比 |
|------|------|------|
| **事件（Event）** | 在什么时机触发 hook | ISR 的中断触发源 |
| **匹配器（Matcher）** | 对什么条件触发（如具体的工具名或命令模式） | ISR 的中断标志位 |
| **动作（Hooks）** | 触发后执行什么（一个或多个命令） | ISR 的中断处理函数 |

Claude Code 支持的 hooks 事件涵盖 30+ 个生命周期事件。对 AE 工程师最实用的：

| 事件类型 | 触发时机 | 嵌入式实用场景 |
|---------|---------|-------------|
| `PreToolUse` | 工具被调用**前** | AI 要修改 CubeMX 生成文件 → 阻止并警告 |
| `PostToolUse` | 工具被调用**后** | 构建成功后 → 自动跑静态分析；git commit 后 → 检查 message 格式 |
| `UserPromptSubmit` | 用户提交消息时 | 用户问代码问题 → 检查项目 CLAUDE.md 是否加载 |

## 配置结构

Hook 配置写在 `.claude/settings.json` 的 `"hooks"` 字段中。基本结构：

```json
\&#123;
  "hooks": \&#123;
    "<事件名>": [
      \&#123;
        "matcher": "<工具名或匹配模式>",
        "hooks": [
          \&#123;
            "type": "command",
            "command": "<你的 shell 命令>"
          }
        ]
      }
    ]
  }
}
```

> ⚠️ **关键细节**：
> - 最外层 key 是 `"hooks"`（复数）
> - 每个事件下的每个条目有一个 `"hooks"` **数组**（不是 `"hook"` 单数对象！）——这是新手最常犯的错误，写 `"hook"` 会导致 hook 被静默忽略
> - `matcher` 区分大小写——`"bash"` 不会匹配 `Bash`
> - 工具输入/输出数据通过 **stdin 以 JSON 格式**传入（不是环境变量），在 shell 命令中通过 `$(cat)` 或 `jq` 解析

### 退出码规则

| 退出码 | PreToolUse 效果 | PostToolUse 效果 |
|--------|----------------|-----------------|
| `0` | 允许继续 | 正常 |
| `2` | **阻止**工具执行 | 无特殊效果 |
| 其他非零 | 非阻塞错误（工具仍执行），stderr 输出到调试日志 | 同左 |

> ⚠️ **重要**：只有退出码 `2` 能真正阻止操作。`exit 1` 不能阻止——工具仍会继续执行。

## 嵌入式场景 1：构建后自动 Lint

**场景**：每次构建成功后自动跑静态分析（`make lint`），避免"编译过了但代码质量差"。

### 配置

```json
\&#123;
  "hooks": \&#123;
    "PostToolUse": [
      \&#123;
        "matcher": "",
        "if": \&#123;
          "command": "make"
        },
        "hooks": [
          \&#123;
            "type": "command",
            "command": "make lint"
          }
        ]
      }
    ]
  }
}
```

> 💡 **说明**：`matcher: ""` 匹配所有工具，但 `"if"` 字段进一步过滤——只在命令包含 `make` 时才触发。这样不会在每次 Bash 调用时都跑 `make lint`。

### 效果

```
你: 帮我编译一下
AI: 运行 make clean && make
    → Build successful.
    [Hook 自动触发]
    → 运行 make lint
    → "i2c_eeprom.c:145: warning: function has 3 return paths but only 2 check error"
    AI: 编译成功。静态分析发现 1 个 warning——i2c_eeprom.c:145 有 3 条返回路径但只有 2 条检查了错误返回。
        要我修复吗？
```

## 嵌入式场景 2：保护 CubeMX 生成的文件

**场景**：用 STM32CubeMX 生成了 `main.c`、`stm32f4xx_it.c` 等文件。这些文件不应该手动修改。Hook 在 AI 即将修改这些文件时阻止操作。

### 配置

```json
\&#123;
  "hooks": \&#123;
    "PreToolUse": [
      \&#123;
        "matcher": "Write|Edit",
        "if": \&#123;
          "file_path": "main\\.c|stm32f4xx_it\\.c|stm32f4xx_hal_msp\\.c"
        },
        "hooks": [
          \&#123;
            "type": "command",
            "command": "echo 'WARNING: 这是 CubeMX 生成的文件，修改会被重新生成覆盖。建议在用户代码区（USER CODE BEGIN...END）内修改，或修改独立的驱动文件。'; exit 2"
          }
        ]
      }
    ]
  }
}
```

> 💡 **说明**：`matcher: "Write|Edit"` 用竖线 `|` 做 OR 匹配——同时匹配 Write 和 Edit 两个工具。`"if"` 的 `file_path` 字段用正则匹配受保护的文件名。退出码 `2` 阻止 AI 执行修改操作。

### 效果

```
你: 帮我在 main.c 里加一个 GPIO 初始化
AI: [准备修改 main.c]
    [Hook 触发，退出码 2 → 阻止]
    WARNING: 这是 CubeMX 生成的文件，修改会被重新生成覆盖。
    建议在用户代码区（USER CODE BEGIN...END）内修改，或修改独立的驱动文件。
    AI: main.c 是 CubeMX 生成的文件。建议我在 bsp/gpio.c 中创建独立的 GPIO 初始化函数，
        然后在 main.c 的 USER CODE BEGIN 区域调用。要我这么做吗？
```

## 嵌入式场景 3：Git 提交后检查 Commit Message

**场景**：团队规定 commit message 格式必须是 `<type>: <subject>`。Hook 在 AI 执行 git commit 后检查格式。

> 💡 **为什么用 PostToolUse 而非 PreToolUse**：PreToolUse 在命令执行**前**触发——这时 commit message 还没创建，无法检查。PostToolUse 在 git commit 完成后触发，可以检查执行结果。

### 配置

```json
\&#123;
  "hooks": \&#123;
    "PostToolUse": [
      \&#123;
        "matcher": "Bash",
        "if": \&#123;
          "command": "git commit"
        },
        "hooks": [
          \&#123;
            "type": "command",
            "command": "LAST_COMMIT_MSG=$(git log -1 --pretty=%B); if ! echo \"$LAST_COMMIT_MSG\" | grep -qP '^(feat|fix|refactor|docs|test|chore):'; then echo 'WARNING: commit message 格式应为 <type>: <subject>。type ∈ \&#123;feat, fix, refactor, docs, test, chore}。当前: '$LAST_COMMIT_MSG; fi"
          }
        ]
      }
    ]
  }
}
```

### 效果

```
你: 提交刚才的修改
AI: git commit -m "update i2c driver"
    [Hook 触发 → 检查最新 commit message]
    WARNING: commit message 格式应为 <type>: <subject>。当前: update i2c driver
    AI: commit message 格式不符合团队规范。改为 "fix: add NACK detection to I2C TXE wait loop" 再提交，可以吗？
```

> ⚠️ **注意**：PostToolUse 在命令执行**后**才触发——上例中 commit 已经创建了，hook 只是发出警告。如果需要**阻止不符合规范的 commit 被创建**，改用 PreToolUse 配合 `"if"` 字段检查命令参数（而非结果）。

## 更复杂的 Hook：用 stdin JSON 精确判断

上面 3 个场景使用了 `"if"` 字段做简单过滤。对于更复杂的判断逻辑（如需要解析工具的具体输入参数），hook 命令可以通过 **stdin** 接收 JSON 数据。

一个接收 stdin 的 PreToolUse hook 示例：

```json
\&#123;
  "hooks": \&#123;
    "PreToolUse": [
      \&#123;
        "matcher": "Edit",
        "hooks": [
          \&#123;
            "type": "command",
            "command": "INPUT=$(cat); FILE=$(echo \"$INPUT\" | jq -r '.tool_input.file_path // \"\"'); if echo \"$FILE\" | grep -qE '(main\\.c|stm32f4xx_it\\.c)'; then echo 'WARNING: 受保护文件 - ' \"$FILE\"; exit 2; fi"
          }
        ]
      }
    ]
  }
}
```

> 💡 `$(cat)` 读取 stdin 中的 JSON → `jq` 提取 `tool_input.file_path` → 判断是否为受保护文件 → 如果是则 `exit 2` 阻止。

> 📖 对于大多数日常场景，场景 1-3 中的 `"if"` 字段已经够用。stdin 方式适用于需要解析工具详细参数的复杂场景。完整参考见 [Claude Code Hooks 官方文档](https://code.claude.com/docs/en/hooks)。

## ⚠️ 常见卡点

| 现象 | 自救方案 |
|------|---------|
| Hook 完全没触发 | ① **检查 `"hook"` vs `"hooks"`**——这是 #1 错误：写了 `"hook": \&#123;...}`（单数+对象）而非 `"hooks": [\&#123;...}]`（复数+数组）。后者是正确格式，前者被静默忽略 ② 确认 hook 写在 `.claude/settings.json` 而不是 CLAUDE.md ③ 重启 Claude Code ④ 检查 JSON 语法（缺少逗号、引号不匹配） |
| Hook 应该阻止操作但没有阻止 | 检查退出码——**只有 `exit 2` 能阻止操作**。`exit 1` 不能阻止（只记录非阻塞错误）。确认你的阻止 hook 用的是 `exit 2` |
| matcher 不生效（匹配不到工具） | ① matcher **区分大小写**——`"bash"` 不匹配 `Bash`，`"edit"` 不匹配 `Edit` ② 使用 `"Write\|Edit"`（竖线分隔）做 OR 匹配 ③ 用 `"if"` 字段做更细粒度的过滤 |
| Hook 命令太复杂，出错难调试 | 把复杂逻辑写成一个独立脚本（如 `.claude/hooks/pre-commit-check.sh`），hook command 只写 `bash .claude/hooks/pre-commit-check.sh`。脚本可以单独测试：`echo '\&#123;"tool_input":\&#123;...}}' \| bash .claude/hooks/pre-commit-check.sh` |
| 更新配置后 hook 不生效 | 重启 Claude Code（hooks 在启动时加载，不支持热更新） |
