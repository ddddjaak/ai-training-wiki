title: Plan Mode & Goal Mode — 先规划，再执行，自动迭代

## 概述

Claude Code 有两个控制"怎么执行"的核心模式。它们互补——一个让你在循环里审方案，一个让 AI 自己在循环里跑到达标。

| | Plan Mode (`/plan`) | Goal Mode (`/goal`) |
|---|-----|-----|
| **你参与度** | 高——你审方案、你确认 | 低——你设条件，AI 自己跑 |
| **适合任务** | 你知道做什么，想看怎么做 | 你知道要达到什么标准 |
| **输出** | 执行方案（你确认后才动手） | 达标结果（AI 自己循环直到达标） |
| **典型耗时** | 2-5 分钟 | 5-30 分钟 |

> **一句话**：`/plan` 是"帮我想清楚怎么做"——你在循环里。`/goal` 是"帮我做到这个标准"——AI 在循环里。

---

## Plan Mode —— 先规划，再执行

## 核心价值

| 没有 Plan Mode | 有 Plan Mode |
|---------------|-------------|
| "帮我实现低功耗管理模块" → AI 直接写代码 → 写到一半发现架构不对 → 推倒重来 | AI 先出方案 → 你审阅确认 → 按确认的方案精确执行 |
| 复杂需求（5+ 文件改动）一次对话塞进去 → AI 顾头不顾尾 | 需求先被拆成清晰的步骤 → 每步可验证 → 不跑偏 |
| 你隐含了大量假设（"当然用 FreeRTOS 啊"）→ AI 不知道 → 写裸机代码 | Plan 阶段强迫你显式化所有假设 → AI 不会误解 |

> **一句话**：Plan Mode 是"先想清楚再动手"的强制开关——和 AE 工程师的核心工作哲学完全一致。

---

## 全景位置

```
Define（理解需求）→ Plan（制定方案）→ Implement（实现）

                    ↑ Plan Mode 作用在这里 ↑
                    
区别：
  interview-me [3-1] → 澄清"客户要什么"
  spec-driven [3-3]  → 写"验收条件是什么"
  Plan Mode          → 规划"怎么一步步做出来"
```

---

## 怎么触发

### 方式一：显式触发

```
/plan
```

进入 Plan Mode 后，AI 不再直接执行代码改动，而是：
1. 分析你的需求
2. 提出实现方案（含步骤、文件改动范围、风险点）
3. 等待你确认或修改
4. 确认后退出 Plan Mode，按方案执行

### 方式二：自然语言触发

```
先帮我规划一下怎么实现这个功能，别急着写代码
```

```
这个改动涉及多个模块，先出个方案我看看
```

### 方式三：自动化触发（CLAUSE.md 配置）

在 CLAUDE.md 中加入以下规则，让 AI 在特定场景**自动**进入 Plan Mode：

```markdown
## Plan Mode 自动触发规则
- 涉及 3 个以上文件的改动 → 必须先 Plan
- 涉及中断 / DMA / 时钟树 的改动 → 必须先 Plan
- 新模块开发 → 必须先 Plan
```

## 做对了的标准

| 成功 | 失败 |
|------|------|
| AI 列出改动方案（受影响文件、修改要点、风险），等你确认 | AI 跳过 Plan 直接改代码——你看到一堆文件被编辑 |
| 你审阅方案时发现一个遗漏 → 口头补充 → AI 更新方案 | 你补充要求时 AI 已经写了一半——改起来比从头写还累 |
| 方案确认后，AI 按方案顺序逐步实现，每步可验证 | AI 一次性改了 5 个文件——你不知道从哪个开始验证 |
| 方案被记录在对话中，30 分钟后还能回溯"为什么选了这个方案" | 写到一半忘了最初的方案是什么 |

---

## 常见误区

| 误区 | 为什么错 | 正确做法 |
|------|---------|---------|
| "小改动不需要 Plan" | 即使是单文件改动，如果逻辑复杂（状态机、多条件分支），Plan 也能帮你提前发现遗漏 | 判断标准不是"文件数量"而是"逻辑复杂度" |
| "Plan 就是让 AI 列个 TODO list" | Plan Mode 产出的是**含风险分析和替代方案**的结构化方案，不是清单 | 要求 AI 在 Plan 中标注"为什么选 A 不选 B"和"最大的风险是什么" |
| "Plan 完了就可以放心让 AI 自己跑" | Plan 是对"要做什么"的共识，但实现细节（函数签名、错误处理）仍需要你关注 | Plan 确认的是方向和步骤，实现过程中仍需审查代码 |
| "Plan Mode 浪费时间" | 对比：无 Plan 写错重来（30+ 分钟）vs 有 Plan 一次到位（Plan 5 分钟 + 实现 15 分钟） | 改动范围越大，Plan 的时间投入回报比越高 |

---

## 与 ae-skills 的配合

| 场景 | 先用 | 再用 |
|------|------|------|
| 客户需求来了 | interview-me（澄清需求） | Plan Mode（出实现方案） |
| 方案方向不明确 | idea-refine（方向对比） | Plan Mode（细化执行） |
| 有现成 spec | spec-driven（写规格） | Plan Mode（拆执行步骤） |
| Plan 输出的步骤太大 | Plan Mode | planning-and-task-breakdown（sub-skill 拆解） |

---

## 常见卡点

| 现象 | 自救方案 |
|------|---------|
| `/plan` 没反应 | 检查 CLAUDE.md 中是否有规则屏蔽了 plan 模式——某些配置可能阻止 AI 切换模式 |
| Plan 太简略（就 2-3 行） | 追问："这个 Plan 不够详细。请列出：受影响文件、每个文件改什么、风险点、替代方案" |
| Plan 太啰嗦（10 页） | "Plan 太长，请压缩到 5 个要点以内，只保留关键决策和最高风险" |
| 执行中偏离了 Plan | "停下。回顾之前的 Plan——第 3 步你改动了不在 Plan 中的文件，为什么？" |
| 执行完了但忘了 Plan 的某些步骤 | 说 "回顾 Plan，还有哪些步骤没完成？"——AI 会对比 Plan 和已完成的改动 |

## Goal Mode —— 设一个目标，AI 自己迭代直到达成

### Plan Mode vs Goal Mode

很多人把这两个搞混。本质区别：

| | `/plan`（Plan Mode） | `/goal`（Goal Mode） |
|---|-----|-----|
| **干什么** | 出一个执行方案，等你确认 | 设一个目标条件，AI 自己循环直到满足 |
| **你参与多少** | 你在循环里——审阅方案 → 确认 → AI 执行 | 你设完目标就退出了——AI 自己修改→验证→检查→再修改 |
| **适合什么** | 单次复杂改动（你知道要做什么，但想先看方案） | 需要多轮迭代的任务（你不知道要改几轮才能达标） |
| **典型时长** | 2-5 分钟（出方案 + 你确认） | 5-30 分钟（AI 自己跑多轮，你去做别的事） |
| **类比** | 建筑师画图给你审 | 自动空调——设 26°C，压缩机自己启停直到达标 |

### AE 工程师什么时候用 Goal Mode

| 场景 | 用 `/plan` | 用 `/goal` |
|------|-----------|-----------|
| 实现一个新功能 | ✅ 先 Plan 再实现 | ❌ 功能需求不是"条件判断" |
| "让所有单元测试通过" | ❌ 你不知道哪些测试会挂 | ✅ 设目标 → AI 逐个修 |
| "消除所有编译警告" | ❌ 你不知道有多少个警告 | ✅ 设目标 → AI 逐个消除 |
| "确保没有函数超过 80 行" | ❌ 你不知道有几个超了 | ✅ 设目标 → AI 扫描→拆分→再扫描 |
| "I2C 通信成功率提高到 99.9%" | ❌ 需要多轮调试 | ✅ 设目标 → AI 反复分析→修改→测试 |

> **判断标准**：如果成功条件可以写成一个**可验证的布尔表达式**（"测试全绿"、"警告数=0"、"成功率>99.9%"），用 `/goal`。如果任务需要人的判断和确认（"方案看起来合理吗？"），用 `/plan`。

### 怎么触发

```
/goal 让当前项目的所有单元测试通过
```

```
/goal 消除 src/ 下的所有编译警告，不能引入新的错误
```

```
/goal I2C 通信成功率在 1000 次测试中达到 100%，当前是 97%
```

### Goal Mode 的工作循环

```
你设目标
  ↓
AI 检查当前状态（离目标还差多少）
  ↓
AI 做一轮修改
  ↓
AI 自己验证（编译/测试/检查）
  ↓
满足目标？── No ──→ 回到"检查当前状态"
  │
  Yes
  ↓
AI 报告：目标达成 + 做了什么
```

> ⚠️ **安全网**：Goal Mode 仍然遵守你的权限配置（2-13）。如果某步操作在 deny 列表中，AI 会停下来等你授权——不会绕过权限。

### 实战示例

```
你：/goal 让 firmware/src/ 下的所有 Warning 清零。不能改功能行为。

AI：[检查] 当前有 14 个 Warning: 8 个 unused variable, 4 个 implicit cast, 2 个 sign compare
  
AI：[第 1 轮]
  修了 8 个 unused variable（删除或加 (void) 抑制）
  编译 → 8 个 Warning 消失，还剩 6 个

AI：[第 2 轮]
  修了 4 个 implicit cast（加显式类型转换）
  编译 → 4 个消失，还剩 2 个 sign compare

AI：[第 3 轮]
  修了 2 个 sign compare（改用正确的有符号/无符号类型）
  编译 → 0 Warning ✓

AI：目标达成！14 个 Warning 全部消除。改动涉及 6 个文件。运行一遍测试确认没有引入回归？
```

### 和 `/loop` 的区别

| | `/goal` | `/loop` |
|---|---------|---------|
| 触发方式 | 设目标条件 | 设时间间隔 + prompt |
| 什么时候结束 | 条件满足就停 | 你手动停（或设次数） |
| 适合什么 | "把这个修好" | "每 5 分钟检查一次 CI" |

### 常见误区

| 误区 | 为什么错 | 正确做法 |
|------|---------|---------|
| "Goal 太模糊 AI 也能理解" | "让代码更好"——AI 不知道"好"的标准是什么 | 目标是可验证的布尔条件："Warning 数=0"、"测试全部通过"、"无函数超 80 行" |
| "设了 Goal 就可以不管了" | AI 可能为了达标做了你不想要的修改（比如删代码来消除 Warning） | 加约束："不能改功能行为"、"不能删除任何函数" |
| "小任务也用 Goal" | 单文件改动不需要 Goal 的循环开销 | 如果一轮就能完成的事 → 直接说，不用 `/goal` |
| "Goal 失败了就是 AI 不行" | 可能是条件本身不可达（比如 84MHz 时钟下生成 1MHz PWM 但 ARR 只有 8-bit） | 先确认目标在技术上可达——AI 失败时会告诉你为什么不可达 |

### Goal Mode 卡点

| 现象 | 自救方案 |
|------|---------|
| AI 循环了 10 轮还没达标 | 目标可能不可达或约束太严。说 "暂停，分析一下为什么还没达标" |
| AI 为了达标删了你的代码 | 加约束："不能删除任何已有功能代码。如果 Warning 无法安全修复，保留并注释说明" |
| Goal 执行中被权限拦截 | Goal Mode 遵守权限配置——被 deny 的操作会暂停等你。调整权限规则后说 "继续" |
| "不知道怎么设目标条件" | 先用自然语言描述你要的结果，AI 会帮你翻译成可验证的条件 |
