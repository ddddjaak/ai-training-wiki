## 核心价值

| 没有 Memory | 有 Memory |
|------------|----------|
| 每次新会话："这个项目用的 MCU 是 STM32F407，HAL 库版本 1.8，跑 FreeRTOS"——你说了第 10 遍 | 新会话开始 → AI 自动从 Memory 加载项目关键信息 → 直接进入正题 |
| 上周分析出一个 bug 根因（"I2C 超时是因为 SCL 被从设备拉低"）→ 这周忘了 → 又问 AI | AI 从 Memory 中检索到上次的分析结论 → 直接引用，不等你重新描述 |
| 团队 3 个人各自跟 AI 对话 → 每个人都要重新教 AI 项目规范 | Memory 是共享的 → 第一个人记录的规范，后面的人自动受益 |

> **一句话**：Memory 是 AI 的"项目笔记"——不是聊天记录的存档，而是结构化知识图谱，AI 会自动在相关场景下检索并引用。

---

## 全景位置

```
CLAUDE.md（静态项目规则）     Memory（动态项目知识）
       ↓                              ↓
   "这个项目用 STM32"          "上次排查发现 I2C 总线上拉电阻
                                从 4.7k 换成 2.2k 解决了噪声问题"
```

区别：
- **CLAUDE.md**：你主动写的、相对固定的项目规范（"用什么芯片、什么 RTOS"）
- **Memory**：AI 自动记录的、持续增长的动态知识（"上次排了什么坑、做了什么决策"）

## 怎么触发

### 方式一：AI 自动记录

当你做了关键决策或发现了重要结论，AI 会自动写入 Memory：

```
你："I2C 超时的根因是 SCL 被从设备拉低，不是总线速率问题"
AI："了解。记录到 memory 中：I2C 超时问题根因是 SCL 被从设备拉低（非速率问题）"
```

### 方式二：显式触发

```
记录一下：这个项目不能用 HAL_Delay()，必须用 FreeRTOS 的 vTaskDelay()
```

```
记住：客户 A 的板子用的是外部 32.768kHz LSE，不是内部 LSI
```

### 方式三：查看已有记忆

```
回忆一下这个项目有哪些关键决策
```

```
这个项目记录过哪些已知 bug？
```

## Memory 的四种文件类型

Memory 文件存储在项目 `.claude/memory/` 目录下，每种类型有不同的用途：

| 类型 | 用途 | 示例 |
|------|------|------|
| `project` | 项目决策、架构选型、已知 bug | "选用 FreeRTOS 而非裸机，因为需要 OTA" |
| `user` | 你的偏好、经验水平 | "熟悉 STM32 HAL 但不熟悉 Zephyr" |
| `feedback` | 你对 AI 工作的反馈 | "上次的代码审查过严——忽略命名建议" |
| `reference` | 外部资源链接、文档引用 | "STM32F407 参考手册下载链接：xxx" |

> Memory 之间可以用 `[[link]]` 互相引用，形成知识网络。

## 做对了的标准

| 成功 | 失败 |
|------|------|
| 新会话开始，问 AI "这个项目用的是什么 MCU"——AI 能回答 | AI 说"你没有在当前对话中告诉过我" |
| 你提到一个 2 周前解决的 bug → AI 自动想起"上次这个问题是 XX 原因" | AI 当成新问题从头分析——完全忘了 2 周前的结论 |
| Memory 里只有**非显而易见的**知识（坑位、决策），不长篇大论复制代码 | Memory 里塞满了代码片段——每次检索都返回无关内容 |
| 团队成员新加入项目 → AI 从 Memory 中自动加载团队规范和已知坑位 | 新人跟 AI 的第一句话永远是"这个项目用的什么芯片" |

## 常见误区

| 误区 | 为什么错 | 正确做法 |
|------|---------|---------|
| "Memory 就是聊天记录" | Memory 是结构化知识图谱，AI 会对内容做提炼和索引——不是原样存储对话 | 让 AI 自动决定记录什么——你只需要在关键时刻显式提醒 |
| "什么东西都往 Memory 里塞" | Memory 越大越慢——非关键信息会稀释检索精度 | 只记录**如果不写下来下次会出错**的信息：决策理由、坑位根因、非标准配置 |
| "Memory 可以替代 CLAUDE.md" | CLAUDE.md 是规则（"必须这样做"），Memory 是知识（"上次这样做是因为…"） | 两者互补：CLAUDE.md 管行为约束，Memory 管知识积累 |
| "Memory 里的东西永远有效" | 项目演进后旧结论可能过时 | 定期清理：问 AI "Memory 中哪些条目已经过时了？" |

---

## 常见卡点

| 现象 | 自救方案 |
|------|---------|
| AI 说"找不到 memory" | Memory MCP server 需要配置——确认 `.claude/mcp.json` 中有 memory server 条目 |
| AI 记录了但新会话不记得 | Memory 依靠语义检索——你的问题要和记录时的用词相关。问法过于宽泛时 AI 检索不到。精确问："项目里 I2C 总线的已知问题是什么？" |
| Memory 太多太杂 | "列出所有 memory，标注哪些已经过时或价值低" → 逐条确认删除 |
| 团队共享的 Memory 别人看不到 | Memory 文件在项目 `.claude/memory/` 目录下——确认已提交到 git |
