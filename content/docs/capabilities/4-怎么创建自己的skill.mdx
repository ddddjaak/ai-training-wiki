## 核心价值

**Before**：你们团队有一套代码审查 checklist——写 ISR 后要检查 5 个点、提交 PR 前要确认 3 个条件。但每次都是靠记忆或者翻文档。新同事不知道这些规则，老人偶尔也会漏。

**After**：把这个 checklist 变成一个自定义 skill。团队成员触发 `/my-team:isr-review` 后，AI 按你们团队的 checklist 逐条审查，输出标准化的结果。规则不再靠记忆——skill 就是活的 checklist。

> **类比**：ae-skills 是"标准库"（通用嵌入式开发流程）。自定义 skill 是你们团队的"内部库"（只适用于你们项目/领域的特定流程）。就像 HAL 给了你通用的 `HAL_I2C_Master_Transmit`，但你还是会封装一个 `BSP_EEPROM_Write` 来处理你们板子上的特定 EEPROM。

---

## 全景位置

```
CLAUDE.md (2-3) ──→ 自定义 skill (2-4) ──→ MCP 插件 (2-5)
       ↑                    ↑
  让 AI 懂项目        把团队流程变成可复用的自动流程
```

CLAUDE.md 是"知识"（告诉 AI 你的项目是什么样的）。自定义 skill 是"流程"（告诉 AI 你的团队怎么做某件事）。

## 怎么触发

创建自定义 skill 不需要触发任何特殊 skill——直接在 `.claude/skills/` 目录下编写 Markdown 文件即可。下面详细讲。

### 两条路：用 skill-creator（推荐） vs 手动编写

创建 skill 有两条路：

| | skill-creator（推荐） | 手动编写 |
|---|-----|-----|
| **怎么做** | 对话式——告诉 AI "我想创建一个 skill 用来 XX"，AI 引导你完成 | 自己写 Markdown 文件 |
| **测试** | 自动生成测试用例，跑 with/without skill 对比 | 自己手动触发验证 |
| **优化** | AI 根据测试结果迭代改进 | 自己看输出、自己改 |
| **适合** | 复杂 skill、团队共享的 skill、需要保证质量的 skill | 简单 skill（< 20 行提示词）、个人临时用 |
| **耗时** | 10-30 分钟（含测试） | 5-15 分钟（不含测试） |

> **推荐**：如果你要创建团队共用的 skill（如 ISR 审查、PR checklist），用 skill-creator——它帮你跑测试，确保 skill 真的有效。如果只是给自己写一个 5 行的快速模板，手动写更快。

## 方式一：用 skill-creator 创建（推荐）

`skill-creator` 是一个专门用来创建和优化 skill 的 skill。它会引导你走完完整的 skill 开发流程——从想清楚 skill 要做什么，到写好 SKILL.md，再到跑测试验证效果，最后迭代优化。

### 触发方式

```
用 skill-creator 帮我创建一个 skill
```

或者直接描述你想创建的 skill：

```
我想创建一个 skill，用来检查嵌入式 C 代码中的中断安全问题。
帮我用 skill-creator 来做。
```

### skill-creator 的完整工作流

skill-creator 会按以下 6 个阶段引导你：

```
1. 捕获意图        → 你想让这个 skill 做什么？什么时候触发？
         ↓
2. 访谈和研究      → AI 追问边界情况、输入格式、输出格式、约束条件
         ↓
3. 编写 SKILL.md   → AI 根据你的回答写出 skill 草稿
         ↓
4. 运行测试        → AI 生成 2-3 个真实测试用例，分别用"有 skill"和"没 skill"跑一遍
         ↓
5. 审查结果        → 你对比有/无 skill 的输出差异，给反馈
         ↓
6. 迭代优化        → AI 根据你的反馈改进 skill → 回到第 4 步 → 直到你满意
```

### 实战演示：创建一个 ISR 审查 skill

以下是使用 skill-creator 的完整过程。

#### 第 1 步：触发并描述需求

```
你：用 skill-creator 帮我创建一个 ISR 审查 skill。
    我们团队的规范：
    - ISR 里不能有 HAL_Delay、delay_us、任何阻塞等待
    - ISR 里不能有 printf/sprintf
    - ISR 不能超过 50 行
    - 直接操作寄存器时不能整个清零（要用 &= ~FLAG）
    - ISR 修改的全局变量必须 volatile
    - 输出格式：[file:line] 违反规则 #N: 描述 + 修复建议
```

#### 第 2 步：AI 追问边界情况

```
AI：几个问题确认一下：
    1. 如果有人写了 while(I2C->SR & FLAG) 这种忙等——算"阻塞操作"吗？
    2. 50 行限制——空行和注释算不算？
    3. Macro 里的寄存器操作要不要检查？
    4. 这个 skill 是全员共享还是只你自己用？

你：1. 算，while 忙等也是阻塞
    2. 不算，只计算实际代码行
    3. Macro 不需要——只检查函数体内的代码
    4. 全团队共享，放项目级别
```

#### 第 3 步：AI 生成草稿并创建测试用例

```
AI：好的，我已经写好 SKILL.md 草稿。趁热打铁——我准备了 3 个测试用例：

    测试 1：一个明显违规的 ISR（包含 delay_us、printf、72 行）
    测试 2：一个完全合规的 ISR（20 行，无阻塞，volatile 正确）
    测试 3：一个边界情况的 ISR（while 忙等，volatile 缺失）

    这 3 个用例能覆盖所有 5 条规则。可以跑测试了吗？
```

#### 第 4 步：AI 并行跑测试

```
AI：正在并行运行测试（有 skill vs 没 skill）...

    eval-0（违规 ISR）：
      有 skill：发现 4 个问题（delay_us, printf, 行数超限, volatile 缺失）
      没 skill：只说"ISR 应该尽量短"（笼统，没给具体行号和修复建议）
    
    eval-1（合规 ISR）：
      有 skill：✅ ISR 审查通过
      没 skill："看起来没什么问题"（没有逐条确认）
    
    eval-2（边界情况）：
      有 skill：发现 2 个问题（while 忙等, volatile 缺失）
      没 skill：完全没发现问题（漏检）
```

#### 第 5 步：你审查结果并给反馈

```
你：测试 0 的结果很好——用 skill 的版本精确找到了具体行号。
    但测试 2 的"没 skill"版本没有逐条确认——不确定它是否真的检查了所有规则。
    
    另外，测试 0 的输出里缺少"修复建议"——skill 找到了 delay_us 但没建议"改为非阻塞计数器"。

AI：收到。我来改进：
    1. 优化输出格式，强制每条发现都要有"修复建议"字段
    2. 加一个"通过标准"——必须逐条确认，不能笼统地说"看起来没问题"
    
    修改好了，重新跑测试？
```

#### 第 6 步：迭代直到满意

```
AI（第 2 轮测试后）：改进后的 skill：
    - 测试 0：现在每条发现都带了修复建议
    - 测试 1：输出"✅ ISR 审查通过，逐条确认："+ 5 条规则的逐条结果
    - 测试 2：2 个问题 + 具体修复建议
    
    还要再优化吗？

你：可以了，这个版本够用了。把最终版写出来吧。

AI：已经把最终版 SKILL.md 写入 .claude/skills/isr-review/SKILL.md。
    团队其他人 clone 项目后，重启 Claude Code 即可使用 /isr-review。
```

### 什么时候应该写测试用例

| 场景 | 要不要测试 | 理由 |
|------|----------|------|
| 团队共享的审查 skill（如 ISR 检查） | ✅ 强烈建议 | 漏检会导致实际 bug 逃逸 |
| 代码生成模板（如 HAL 初始化） | ✅ 建议 | 要确保生成代码符合团队规范 |
| 输出格式固定的 skill | ✅ 建议 | 格式错了后面的自动化流程就断了 |
| 个人用的快速提问模板 | ❌ 不需要 | 你自己看一眼就知道对不对 |
| 纯主观输出的 skill（如"帮我把技术点写得更易懂"） | ❌ 通常不需要 | 效果好坏靠人判断，测试用例难覆盖 |

## 方式二：手动编写 Skill

如果你选择手动编写（简单 skill 或你更喜欢直接写 Markdown），以下是格式参考。

## Skill 的基本结构

每个 skill 是一个 Markdown 文件，由四部分组成：

```
1. 位置（放哪个目录）
2. 元数据（frontmatter —— 名称、描述、触发方式）
3. 提示词（系统提示 —— 告诉 AI 这个 skill 要做什么）
4. 验证（怎么确认 skill 生效了）
```

### 目录结构

Skill 文件放在项目或用户的 `.claude/skills/` 目录下：

```
项目级别（只在这个项目里生效）:
你的项目/.claude/skills/my-skill/SKILL.md

用户级别（在所有项目里生效）:
~/.claude/skills/my-skill/SKILL.md
```

> 💡 **选择原则**：团队共用的 skill（如代码审查 checklist）放项目级别。个人偏好的 skill（如你的特定提问模板）放用户级别。

### Skill 文件模板

下面是一个完整的 skill 示例——**ISR 审查 skill**（芯片工程师的真实场景）：

```markdown
---
name: isr-review
description: 按团队规范审查中断处理函数 —— 5 条红线检查
---

# ISR 审查

按以下 5 条团队红线逐条审查给定的中断处理函数。

## 审查规则

对于每个 ISR 函数，逐条检查：

1. **无阻塞操作**: ISR 中不能有 `delay_us`、`HAL_Delay`、`while(...)` 忙等循环。如果发现 → 建议改为非阻塞计数器。
2. **无 printf**: ISR 中不能有 `printf` 或 `sprintf`（阻塞 + 栈溢出风险）。如果发现 → 建议改为标志位 + 主循环打印。
3. **长度 < 50 行**: 如果 ISR 超过 50 行 → 建议将处理逻辑提取为独立函数。
4. **直接操作寄存器时正确清除标志位**: 如果使用 `REG = 0` 清除标志 → 建议改为 `REG &= ~FLAG`。
5. **全局变量使用 `volatile`**: 如果 ISR 修改的全局变量未声明为 `volatile` → 标注缺失。

## 输出格式

对于每个发现，按以下格式输出：

```
[file:line] 违反规则 #N: 具体描述 + 修复建议
```

如果没有发现任何问题，输出 `✅ ISR 审查通过，未发现违反 5 条红线的问题。`
```

### 逐段解释

#### frontmatter（元数据）

```yaml
---
name: isr-review              # skill 名称（用于触发: /my-team:isr-review）
description: ...              # 一句话描述（用于 skill 列表展示）
---
```

#### 系统提示（正文）

Markdown 正文是 AI 的系统提示词——告诉 AI 在这个 skill 被触发后应该怎么工作。写作原则：
- **用祈使句**：直接告诉 AI 做什么（"逐条检查"、"输出"、"标注"），不要用"你应该"这种弱语气
- **指定输出格式**：让 AI 输出结构化内容，不是自由发挥
- **定义"通过"标准**：告诉 AI 什么情况算"没有问题"

## 嵌入式团队常用 Skill 示例

以下 3 个 skill 是你可能马上想创建的——直接从下面的模板复制，按你的团队规范微调。

### Skill 1: PR 提交前自查

```markdown
---
name: pr-checklist
description: 提交 PR 前的自动检查清单
---

# PR 提交前审查

对当前改动做以下检查。每一项输出 ✅（通过）或 ❌（未通过 + 修复建议）。

## 检查项

1. **编译**: 确认 `make clean && make` 零错误零警告
2. **ISR 红线**: 改动中是否涉及中断处理函数？如有 → 执行 ISR 5 条红线检查
3. **动态内存**: 改动中是否出现 `malloc`/`free`？如有 → ❌（项目禁用动态内存）
4. **volatile 修饰**: 被中断或 DMA 修改的全局变量是否声明为 `volatile`？
5. **错误处理**: 所有 HAL 函数返回值是否被检查？
6. **commit message**: commit message 是否按 `<type>: <subject>` 格式？

## 输出格式

```
PR 审查结果:
  编译: ✅ / ❌ (说明)
  ISR 红线: ✅ / ❌ (说明)
  动态内存: ✅ / ❌ (说明)
  volatile: ✅ / ❌ (说明)
  错误处理: ✅ / ❌ (说明)
  commit message: ✅ / ❌ (说明)

总结: X/6 项通过
```
```

### Skill 2: HAL 外设初始化模板

```markdown
---
name: hal-init
description: 生成符合团队规范的 HAL 外设初始化代码
---

# HAL 外设初始化代码生成

当用户请求生成外设初始化代码时，按以下模板生成。

## 模板结构

```c
/**
 * @brief  [外设名] 初始化
 * @param  [配置参数]
 * @retval HAL_StatusTypeDef
 */
HAL_StatusTypeDef BSP_[外设名]_Init([参数]) \&#123;
    /* 1. 时钟使能 */
    __HAL_RCC_[外设时钟]_CLK_ENABLE();

    /* 2. GPIO 配置（如果需要） */
    // GPIO_InitTypeDef gpio = \&#123;0};
    // ...

    /* 3. 外设基础配置 */
    [外设]_HandleTypeDef h[外设] = \&#123;0};
    h[外设].Instance = [外设基地址];
    h[外设].Init.[参数] = [值];
    // ...

    /* 4. 初始化 */
    if (HAL_[外设]_Init(&h[外设]) != HAL_OK) \&#123;
        return HAL_ERROR;
    }

    /* 5. 中断配置（如果需要） */
    // HAL_NVIC_SetPriority([IRQn], [优先级], 0);
    // HAL_NVIC_EnableIRQ([IRQn]);

    return HAL_OK;
}

/**
 * @brief  [外设名] 反初始化（进入低功耗前调用）
 */
HAL_StatusTypeDef BSP_[外设名]_DeInit(void) \&#123;
    /* 关闭外设时钟（可选，用于低功耗） */
    // __HAL_RCC_[外设时钟]_CLK_DISABLE();
    return HAL_OK;
}
```

## 约束
- 所有外设句柄声明为 `static`（模块内可见）
- 函数前缀 `BSP_`（板级支持包）
- 必须包含初始化失败的错误返回路径
- 必须包含 DeInit 函数（用于低功耗场景）
```

### Skill 3: 日志调试辅助

```markdown
---
name: debug-log
description: 在函数关键路径插入调试日志
---

# 调试日志插入

在指定函数的以下位置插入调试日志（用项目日志宏 `LOG_DBG`）：

## 插入位置
1. 函数入口：打印函数名 + 关键入参值
2. 每个 `return` 前：打印返回值 + "退出原因"
3. 每个错误分支：打印错误码 + 上下文

## 日志格式
```c
LOG_DBG("[模块名] 函数名: 消息 key=value");
```

## 约束
- 使用项目统一的 `LOG_DBG` 宏（定义在 `config/debug_config.h`）
- 不在 ISR 中插入日志
- 不在时间敏感的路径插入（如 DMA 完成回调的前 5 行）
- 所有日志语句用 `#ifdef DEBUG_ENABLED` 包裹（如果项目有要求）
```

## 怎么验证 Skill 生效了

### 方法 1：直接触发

创建 skill 后，在 Claude Code 中触发它：

```
/[skill-name]:your-skill
```

如果 AI 按你的提示词执行了 → ✅ skill 生效。

### 方法 2：测试场景

用一个你知道正确答案的测试输入触发 skill，检查输出是否符合预期：

```
/isr-review: 审查 src/timer_irq.c 的中断处理函数
```

如果输出包含 5 条红线的逐条检查 → ✅ 输出格式正确。
如果输出中有你期待的发现（比如你知道第 42 行有个 `delay_us`）→ ✅ 审查内容正确。

### 方法 3：错误输入测试

给一个故意违反规则的输入，看 skill 能否捕获：

```
/hal-init: 生成一个 I2C 初始化函数，但不要错误处理路径
```

如果 AI 拒绝（"按团队规范，初始化函数必须有错误返回路径"）→ ✅ skill 约束被遵守。

## ⚠️ 常见卡点

| 现象 | 自救方案 |
|------|---------|
| 触发 skill 后 AI 没有按提示词执行 | ① 检查 SKILL.md 文件名和路径是否正确（区分大小写）② 确认 frontmatter 格式正确（`---` 前后无空格）③ 重新启动 Claude Code（skill 在启动时加载） |
| skill 名称冲突（和已有 skill 同名） | 使用命名空间前缀：`name: myteam-isr-review` → 触发 `/myteam-isr-review`。不要用 `ae-skills:` 前缀——那是保留给官方 skill 的 |
| skill 提示词太长或太短 | 太短（< 10 行）→ AI 不知道具体要做什么。太长（> 100 行）→ 挤占对话上下文。有效 skill 提示词通常在 20-60 行 |
| 别人看不到我创建的 skill | 如果放在用户级别（~/.claude/skills/）→ 只有你能用。团队共享的 skill 必须放项目级别（项目的 .claude/skills/）并提交到 git |
