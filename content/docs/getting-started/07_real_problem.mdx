# 用 Claude Code 解决真实问题

## 概述

你已经装好了 Claude Code，知道了怎么对话、怎么用命令、怎么管权限。现在把它们全部串起来：**从头到尾用 Claude Code 解决一个真实的嵌入式 Bug。**

> 这一篇是关键。不是"看"，是"跟着做"。打开终端，cd 到你的项目，跟着下面的步骤一步步操作——你会用 Claude Code 跑完从"发现 bug"到"修复并验证"的完整链路。

> ⚠️ **关于时间**：下面的完整流程约 15-20 分钟。实际操作中，简单 bug 可能更快，复杂 bug（需要硬件辅助调试）会更长。重点不是速度——是让你看到一个完整的 AI 辅助调试闭环。

---

## Bug 场景：I2C NACK 导致系统死锁

**背景**：你维护一个通过 I2C 读写外部 EEPROM 的驱动模块。最近 QA 报告了一个问题——在 EEPROM 偶尔不响应时，整个系统卡死。用调试器连接后发现，PC 指针卡在 I2C 事件中断处理函数 `I2C1_EV_IRQHandler` 内部。

**初步分析**：当 EEPROM 从机返回 NACK（不应答）时，I2C 驱动的等待循环没有检查 NACK 标志，一直死等一个永远不会置位的标志位——而这段代码跑在中断上下文里，导致同优先级及以下中断全部被阻塞。

复现方式：拔掉 EEPROM 模块的 I2C 线（MCU 侧有上拉电阻时 I2C 线会被拉高，从机无响应表现为 NACK），系统必死。

## 完整链路：定位 → 修复 → 验证

### 步骤 1：让 AI 了解你的问题（2 分钟）

**准备工作**：cd 到项目目录，启动 `claude`。

```
$ cd ~/projects/chip-firmware
$ claude
```

**你的第一条消息**：
```
我们有一个 I2C 读写 EEPROM 的驱动模块。当 EEPROM 从机不响应（NACK）时，
整个系统卡死。用调试器确认 PC 卡在 I2C1_EV_IRQHandler 中断处理函数内部。
请帮我：
1. 找到 I2C EEPROM 相关的所有源文件
2. 定位发送/读写函数中所有 while 循环等待硬件状态位的代码
3. 检查这些等待点：是否检查了 NACK（AF）标志？是否有超时退出？
```

AI 会先用 Glob 找文件、用 Grep 搜 `while` 循环和 I2C 状态寄存器访问，然后列出各个等待点。

### 步骤 2：AI 定位问题代码（2-3 分钟）

AI 用 Grep 搜索 `I2C1->SR1` 相关的等待循环，输出类似：
```
i2c_eeprom.c 中找到 3 处等待 I2C 状态位的 while 循环：

1. 第 145 行 — 等待 TXE（发送寄存器空）
   while (!(I2C1->SR1 & I2C_SR1_TXE));      ← ❌ 无超时，无 AF 检查

2. 第 172 行 — 等待 BTF（字节传输完成）
   uint32_t timeout = 100000;
   while (!(I2C1->SR1 & I2C_SR1_BTF) && --timeout);  ← ✅ 有超时

3. 第 201 行 — 等待 STOP 条件完成
   while (!(I2C1->SR1 & I2C_SR1_STOPF));     ← ❌ 无超时，无 AF 检查
```

**关键发现**：第 145 行和第 201 行**既没有 NACK 检查，也没有超时退出**。当从机 NACK 时，`TXE` 和 `STOPF` 永远不会置位——代码在这个 while 循环里死锁，而它跑在中断上下文（`I2C1_EV_IRQHandler`）里，同优先级及以下中断全部被阻塞，包括 RTOS 调度器（如果使用）。

> ⚠️ **注意**：上面 AI 的输出是理想情况下的结果。在你的实际项目中，AI 可能不会在第一次就给出这么精确的分析——你可能需要追问或补充上下文（如"这是 STM32F4，用的是 HAL 库"）。这不是 AI 的失败——这是正常的人机协作。调试的本质是迭代。

### 步骤 3：AI 深入分析（2 分钟）

**你继续问**：
```
检查第 145 行的 I2C_WriteByte 函数：
1. 这个函数是在什么上下文里被调用的（中断还是线程）？
2. NACK（AF 标志）应该怎么检测和处理？
3. 超时退出后，怎么恢复 I2C 总线让后续通信能继续？
```

AI 会追溯调用链：
```
I2C1_EV_IRQHandler (中断)
  → I2C_HandleTXE_IRQ (i2c_master.c:88)
    → I2C_WriteByte (i2c_eeprom.c:140)
      → while (!(I2C1->SR1 & I2C_SR1_TXE));  ← 卡在这里
```

确认：在中断上下文里死等。AI 进一步指出：

- **主防线缺失**：代码没检查 `I2C_SR1_AF`（Acknowledge Failure）标志。NACK 发生后 AF 立即置位，应该优先检查 AF 并立即退出，而不是继续等 TXE。
- **次防线缺失**：没有重试计数器。即使 AF 检查是主防线，也应该有一个非阻塞的重试上限作为兜底。
- **设计原则**：中断处理函数中**不应有阻塞延迟**（如 `delay_us()`）。ISR 的超时应该用计数器（每次 ISR 进入 +1），而不是忙等。

### 步骤 4：AI 修复代码（3 分钟）

**你说**：
```
帮我修复 i2c_eeprom.c 第 145 行：
1. 先检查 AF（NACK）标志——如果置位，清除 AF，发送 STOP，记录错误，退出
2. 添加非阻塞重试计数器（每次进入 ISR +1），超过上限后同样发 STOP 退出
3. 错误信息通过全局标志 g_i2c1_error 通知应用层
4. 第 201 行同样处理（AF + 计数器）
```

> 💡 **为什么这样修**：你的指令给了 AI 正确的设计约束——AF 检查是主防线、非阻塞计数器是兜底、不在 ISR 里放 delay。如果你是新手还不确定怎么修，可以先问 AI "修复这个 bug 的最佳实践是什么"，让它先给你讲方案，你再审。

AI 生成修改——弹出确认框，你看到具体 diff：

```diff
- while (!(I2C1->SR1 & I2C_SR1_TXE));
+ /* 主防线: 检查 NACK (AF) 标志 */
+ if (I2C1->SR1 & I2C_SR1_AF) \&#123;
+     I2C1->SR1 &= ~I2C_SR1_AF;          /* 清除 AF 标志 */
+     I2C1->CR1 |= I2C_CR1_STOP;         /* 发送 STOP */
+     g_i2c1_error = HAL_ERROR;           /* 通知应用层 */
+     return;
+ }
+ /* 次防线: 非阻塞重试计数器 */
+ if (++txe_retry_count > I2C_TIMEOUT_MAX) \&#123;
+     I2C1->CR1 |= I2C_CR1_STOP;
+     g_i2c1_error = HAL_TIMEOUT;
+     return;
+ }
+ if (!(I2C1->SR1 & I2C_SR1_TXE)) \&#123;
+     return;  /* 还没就绪, 下次 ISR 触发再检查 */
+ }
+ txe_retry_count = 0;  /* 重试成功, 重置计数器 */
```

同时也需要给第 201 行打同样的补丁，并在文件顶部声明 `g_i2c1_error` 和 `txe_retry_count`。

**你审查 diff**：确认逻辑正确——AF 优先检查 → 非阻塞计数兜底 → 错误通过全局变量通知应用层 → ISR 中无阻塞延迟。确认后接受修改。

> **你的角色**：AI 帮你写了修复代码，但**你审查了 diff**。你确认了：AF 检查是主防线、没有在 ISR 里放 `delay`、错误通知路径通顺。AI 是执行的，你是决策的。

### 步骤 5：编译验证（2 分钟）

**你说**：
```
帮我编译项目，检查修改后是否能编译通过
```

AI 运行 `make`（或你的构建命令）。

**现实情况**：首次编译可能会报几个 warning 或错误——比如你忘了声明 `I2C_TIMEOUT_MAX` 宏、`g_i2c1_error` 的类型和其他模块冲突。这很正常。AI 会根据编译输出继续调整：

```
> 编译报错了: g_i2c1_error 未声明，I2C_TIMEOUT_MAX 未定义

AI: 我来修复——

- 在 i2c_eeprom.h 中添加 extern volatile HAL_StatusTypeDef g_i2c1_error;
- 在 i2c_eeprom.h 中添加 #define I2C_TIMEOUT_MAX 1000
```

几次迭代后编译通过：
```
[CC] i2c_eeprom.c
[LD] firmware.elf
Build successful.
```

### 步骤 6：验证逻辑（3 分钟）

**你说**：
```
帮我检查修改后的代码：
1. 正常路径（从机正常应答）——TXE 正常置位，AF 和计数器不触发
2. NACK 路径（从机不应答）——AF 标志置位 → 立即退出，不阻塞
3. 异常路径（从机既不 ACK 也不 NACK，如总线浮空）——计数器超时后退出
```

AI 逐路径分析修改后的代码流，画状态图或文字推理。你不需要真的上硬件跑——先做**纸上验证**，确认逻辑覆盖了三种路径。

> 💡 这个步骤在真实的嵌入式调试中非常重要。在你能上硬件重现 bug 之前，先在代码层面验证修复逻辑的正确性，可以省去很多烧录-等待-崩溃的循环。

**附加步骤（如果有硬件环境）：**

如果你想跑真实的硬件测试：
```
> 帮我把 I2C 的 SCL 引脚配置为 GPIO 输出，写一个手动发 9 个时钟脉冲的总线恢复函数，
  当 g_i2c1_error != HAL_OK 时由应用层调用，恢复总线后重试通信
```

### 步骤 7：你学到了什么（1 分钟）

回顾你刚才做了什么：

```
描述 Bug + 调试器线索 (步骤1)
→ AI 搜索代码，定位所有等待点 (步骤2)
→ AI 分析调用链 + 指出缺失的 AF 检查 (步骤3)
→ 你给出设计约束，AI 生成修复 (步骤4)
→ 编译迭代直到通过 (步骤5)
→ 纸上验证三种路径 (步骤6)
→ Bug 从定位到修复，完整闭环
```

**这就是一个完整的调试工作流。** 在 Claude Code 里，这个流程有一个专门的名字——

> **`debugging-and-error-recovery` skill**
>
> 在下一篇模块（M2），你会看到 Claude Code 内置了一套完整的技能体系——调试、审查、测试、增量开发、安全审查……刚才你手动走完的每一步，这些技能可以帮你做得更系统、更自动化。
>
> 但现在，你已经知道了核心循环：**描述问题 → AI 定位 → AI 修复 → 你审查 → 编译/测试验证**。这就是日后你每天用来提效的工作模式。

## 你自己试试

找一个你项目里的真实 bug——或者至少是一个你想改进的地方。打开 Claude Code，从头到尾走一遍上面的链路。

**目标**：自己跑通一次"描述 → 定位 → 修复 → 验证"的完整链路。成功标准：AI 帮你找到了具体代码位置，并给出了能编译通过、逻辑正确的修复。

> ⚠️ **实用起点**（如果你还没想到从哪里开始）：
> - "搜索项目中所有的 while(1) 空循环，检查哪些没有超时退出"
> - "找到所有未检查返回值的 HAL 函数调用——遗漏返回值可能导致静默失败"
> - "检查所有中断处理函数中是否有超过 10 行的代码（ISR 应该尽量短，长代码应提取为独立函数）"
> - "找到所有直接操作外设寄存器的代码，检查是否有遗漏的 volatile 修饰"
