# 权限与沙箱

## 概述

Claude Code 能读你的文件、改你的代码、跑你的终端命令——**但它做任何有副作用的操作之前，都必须经过你确认。** 这是 Claude Code 的沙箱机制：AI 被关在一个"围栏"里，跨过围栏之前，你必须点头。

> **类比**: 把 AI 想象成一个在实验室里的实习工程师。他可以在工作台上随便看资料、做笔记（读文件、搜索），但要动实验设备（改代码、跑命令）之前，他必须喊你过来确认——"我准备把这条线接上，你看了说行我再动。"

---

## 三种权限模式

Claude Code 的权限不是"开"或"关"的二元开关，而是三种模式：

| 模式 | AI 行为 | 适用场景 |
|------|---------|---------|
| **Ask**（默认） | 每次操作前弹确认框，你选 Allow / Deny | 大部分日常操作——改代码、跑测试、git 操作 |
| **Always Allow** | 这类操作以后自动通过，不再弹窗 | 你信任的重复性操作——如 `git status`、`ls`、读文件 |
| **Deny** | 这类操作以后自动拒绝，不再弹窗 | 你不想 AI 碰的操作——如 `rm -rf`、`git push --force` |

当你看到这样的确认框时：

> 📷 **截图**：Claude Code 权限确认弹窗（待补充截图）

- 点 **Allow** = 这次允许，下次同样操作还会问
- 点 **Always Allow** = 这次允许 + 以后同样操作都不再问（规则写入 settings.json）
- 点 **Deny** = 这次拒绝（下次还会问）
- 点 **Deny All** = 这次拒绝 + 以后同样操作都自动拒绝

## 推荐权限配置（嵌入式项目）

以下配置是嵌入式项目的推荐基线，可以直接复制到你的 `~/.claude/settings.json` 中：

```json
\&#123;
  "permissions": \&#123;
    "allow": [
      "Bash(git:*)",
      "Bash(grep:*)",
      "Bash(find:*)",
      "Bash(ls:*)",
      "Bash(cat:*)",
      "Bash(make:*)",
      "Bash(cmake:*)",
      "Bash(arm-none-eabi-*:*)" 
    ],
    "deny": [
      "Bash(rm:-rf*)",
      "Bash(chmod:777*)",
      "Bash(git:push*--force*)",
      "Bash(git:reset*--hard*)"
    ]
  }
}
```

**逐条解释**：

| 规则 | 为什么 |
|------|--------|
| `git:*` always-allow | `git status`、`git diff`、`git log` 都是只读的，每次弹窗太烦 |
| `grep:*` / `find:*` / `ls:*` / `cat:*` always-allow | 纯读操作，零副作用 |
| `make:*` / `cmake:*` always-allow | 编译你本来就经常跑，不用每次都确认 |
| `arm-none-eabi-*:*` always-allow | ARM 工具链命令（如 `arm-none-eabi-gcc`、`arm-none-eabi-objdump`）是日常操作 |
| `rm -rf` deny | 递归强制删除——AI 永远不能碰 |
| `chmod 777` deny | 把所有文件权限全放开——危险操作 |
| `git push --force` deny | 强制推送——覆盖远程历史 |
| `git reset --hard` deny | 硬重置——丢失本地修改 |

> **你自己的项目**：把编译器前缀替换成你用的（如 `riscv64-unknown-elf-*`、`xtensa-esp32-*`）。允许列表加得越多，日常弹窗越少；deny 列表是安全底线。

---

## 用 `/permissions` 管理规则

不用手动编辑 JSON——用命令更方便：

```bash
> /permissions
```

会列出当前所有权限规则，支持交互式添加/删除。比手改 JSON 更不容易出错。

---

## ⚠️ 弹窗太多怎么办

如果每次操作都在弹窗，说明你的 allow 列表没配置好。三步自救：

**步骤 1**：先用 `/permissions` 查看当前规则。

**步骤 2**：识别反复弹窗的操作。比如你发现每次 `git status` 都在弹窗——就在弹窗时点「Always Allow」，或手动加规则：
```json
"Bash(git:status)": "always-allow"
```

**步骤 3**：把常见的只读命令和工具链命令加入 allow 列表，把危险操作加入 deny 列表。

**原则**：**只读操作 = always-allow，写入操作 = 按信任度决定，破坏性操作 = deny。**

## 常见问题

**Q: 我不小心点了 Always Allow，怎么撤销？**

输入 `/permissions`，找到对应规则删除或改为 Ask。

**Q: Always Allow 后 AI 会不会偷偷改我的代码？**

不会。Always Allow 只是跳过弹窗，但 AI 的每次修改你都能在终端里看到。如果不同意，在对话里说"不对，改回去"即可。加上你在用 git，`git diff` 能清晰看到所有改动。

**Q: 我能不能全部 Allow？**

技术上可以，但不建议。至少把 `rm -rf`、`git push --force`、`chmod 777` 放在 deny 列表。这些操作的破坏力太大，AI 不应该有一键执行的权限。

**Q: 权限配置是全局的还是按项目的？**

`~/.claude/settings.json` 是全局的，对所有项目生效。你也可以在项目目录下的 `.claude/settings.json` 中覆盖特定项目的权限规则（项目级优先于全局）。
